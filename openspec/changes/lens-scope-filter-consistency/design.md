## Context

`OverviewPage.svelte` renders a scope filter for each populated entity card (repos for Changes, projects for Issues, feeds for Articles). All three already share one code path: `CHIP_THRESHOLD = 5`; `≤5` visible facet values render a row of toggle `Chip`s (one per value), `>5` renders a `MultiSelect` (`variant="chip"`) with an "All X ⌄" trigger and a searchable checkbox overlay. The guard for showing the row at all is `visX.length > 0` for all three axes (`visRepos`, `visProjects`, `visFeeds`).

Observed problem: with exactly one facet value (a lens with a single connected repo, project, or feed), the row still renders — one `Chip` whose label is the raw value. Repo/project identifiers can be long (`github.com/org/very-long-repo-name`), and nothing bounds a chip's width inside `.scope-filter` (`display: flex; flex-wrap: wrap`, no `flex-basis`/`max-width`), so `Chip`'s own `overflow: hidden; text-overflow: ellipsis` never engages — the chip just grows to fit the text. This reads as visual noise (a single-value filter offers no real choice) and, on long values, as a layout inconsistency next to the compact "All feeds ⌄" trigger shown when there are many values.

Second observed problem, in the `>5` `MultiSelect` branch: "show everything" is reachable two ways that look different and can silently diverge. Never touching the filter leaves `values: []` — the spec's documented "empty means show everything" state, which is inherently dynamic (a value that appears later is automatically included). Clicking the `MultiSelect`'s own "Select all" button (`MultiSelect.svelte:134-136`, `selectAll()`) instead calls `onchange([...new Set([...values, ...enabledValues])])` — it snapshots every *currently enabled* option into an explicit array. The trigger then reads e.g. "9 selected" instead of "All feeds," and if a 10th feed appears in a later fetch, that lens silently excludes it even though the user's intent when clicking "Select all" was "show everything." `MultiSelect.svelte` cannot simply be changed to make `selectAll()` clear to `[]` globally: `LensEditor.svelte` (`sidebar/LensEditor.svelte:333-357`) also consumes `MultiSelect`, but there empty means "no lens members chosen yet" (the opposite semantic) — for that consumer, `selectAll()` populating an explicit list of every connected source is the correct, intended behavior. The fix must therefore live in the caller (`OverviewPage.svelte`), not the shared primitive.

## Goals / Non-Goals

**Goals:**
- Hide the scope-filter row when a facet has fewer than 2 visible values, for all three axes (repos, projects, feeds), consistently.
- Bound chip width and truncate with a tooltip for the 2–5 count range so long values never stretch the row.
- Collapse the `>5` `MultiSelect`'s "every value selected" state to the same stored filter (`[]`) as "never filtered," for all three axes, so there is exactly one "show everything" state and it always stays inclusive of values that appear later.

**Non-Goals:**
- Changing the `>5` vs `≤5` chip/`MultiSelect` render threshold itself.
- Changing `Chip.svelte` or `MultiSelect.svelte` themselves — both the truncation fix and the select-all collapse are local to `OverviewPage.svelte`'s markup/handlers, not the shared primitives (see the `LensEditor.svelte` conflict below for why `MultiSelect.svelte` itself is out of scope).
- Any change to filter *semantics* beyond this collapse (facet derivation, `applyLensFilter`, `deriveLensFacets`, the type/entities axis) — untouched.

## Decisions

- **Row visibility threshold: `> 0` → `> 1`.** A single-value facet offers no filtering choice (toggling it off would just hide everything), so showing it as a no-op control is pure noise. Raising the floor to 2 removes the row entirely in the single-value case — the most direct fix for the long-single-chip problem, and it needs no new state or component. Alternative considered: keep showing it but visually de-emphasize (e.g., render as inert text) — rejected, adds a fourth visual state for zero functional benefit.
- **Truncate via a bounded `max-width` on the chip element within `.scope-filter`, not inside `Chip.svelte`.** `Chip.svelte` is a shared primitive used elsewhere (e.g. the issue-priority chip) where the label is always short — capping width there would risk silently truncating unrelated call sites without justification. Bounding it locally in `OverviewPage.svelte`'s `.scope-filter` context (targeting the chip wrapper) keeps the constraint scoped to where the long-label problem actually occurs. A `title={value}` attribute is added alongside the `Chip` for the full value on hover, matching the truncation-plus-tooltip pattern already used for `.row-title` elsewhere on this page (see `OverviewPage.svelte` row title styling — no separate tooltip primitive needed since native `title` already carries this job on this page).
- **Threshold width ~14–16rem.** Long enough that the large majority of repo/project/feed names show in full, short enough to keep the row visually compact next to the `MultiSelect` trigger's width. Not derived from a token (no existing token models "chip max width"); a plain CSS value local to this component, consistent with how other one-off layout constraints in this file (e.g. `.scope-picker { width: min(22rem, 100%); }`) are already handled inline rather than tokenized.
- **Collapse full-selection to `[]` in `OverviewPage.svelte`'s `onchange` handlers, not in `MultiSelect.svelte`.** Each axis's `onchange` already has access to its own `visX` (the full known facet set for that axis, e.g. `visRepos`). Since `MultiSelect`'s `options` are always derived from that same `visX`, an incoming selection can only ever be a subset of it — so `vals.length >= visX.length` (equivalently `vals.length === visX.length`, since duplicates aren't possible from a listbox) reliably means "every currently-known value is selected," regardless of whether the user got there via the "Select all" button or by manually checking every box one at a time. On that condition, the handler stores `[]` instead of `vals`. This deliberately treats "manually checked everything" the same as "clicked Select all" — both mean the same thing to the user, so both should mean the same thing to the filter. Alternative considered: add a new `MultiSelect` prop (e.g. `selectAllBehavior: 'explicit' | 'clear'`) so `selectAll()` itself could branch per-consumer — rejected as unnecessary API surface on a shared primitive when the same effect is achievable in three lines at each call site, and it would leave the "manually check every box" path still snapshotting (the prop only intercepts the button, not organic full selection).

## Visual language

No new colour, motion, or hierarchy — this is a scope reduction (hiding a redundant control) and a containment fix (bounding an existing chip's width), not a new visual element. The truncated chip keeps `Chip.svelte`'s existing toggle-pill appearance and `selected`/hover/focus states unchanged; the only addition is the native `title` tooltip (browser-default, no custom styling) and a `max-width` that lets the existing `overflow: hidden; text-overflow: ellipsis` in `Chip.svelte` actually take effect. Reduced-motion and WCAG-AA contrast are unaffected since no new interactive states or colours are introduced.

## Risks / Trade-offs

- [A facet legitimately has exactly 1 value today but grows to 2+ later in the same session] → No special handling needed: the guard is reactive (`$derived`), so the row appears automatically once the facet count crosses the threshold; no stale state.
- [Hiding the single-value row removes the only visible indicator of which repo/project/feed a lens is scoped to] → Acceptable: the repo/project/feed name is already visible per-row in each Change/Issue/Article's own metadata line (e.g. `changeMeta`, `feedLabel`), so removing the redundant filter chip doesn't remove the information, only the (useless) control.
- [`title` tooltips are not accessible to touch-only users] → Acceptable: this only affects the fallback for a truncated 2–5-value label; the full value is always available by opening any item in that scope, and this matches existing truncation+title usage elsewhere on the page.
- [A user deliberately wants "exactly these N values, and never include anything added later" — a pinned snapshot, distinct from "everything"] → Not supported after this change, for the `>5` scope-filter axes. Accepted: nothing in the `lenses` spec establishes this as a desired capability today (the existing "union of present + selected" language is about not stranding a now-absent selected value, not about pinning a snapshot against future growth), and the ambiguity/staleness this collapse fixes is worse than the loss of an unrequested capability.
- [The `vals.length >= visX.length` check runs on every `onchange`, including ordinary partial-selection changes] → Negligible cost: `visX` is already a `$derived` array in scope; the comparison is an O(1) length check, not a full equality scan.

## Open Questions

None — scope is narrow and fully resolved by the decisions above.
