# Smart folders — live GitLab MRs in the pinned section

## Why

The pinned section can keep the *places* you return to, but not the *work that finds you*: merge requests assigned to you, MRs waiting on your review. Today that list lives in a GitLab tab you re-open and refresh all day. This change puts it inside the Space — a **smart folder** in the pinned tree whose children are live query results from GitLab (v1 connector), each row one click from the MR with its pipeline status at a glance, refreshed automatically. Direct user-visible value (shape a): "my review queue, one glance away, inside the Space it belongs to" — with self-hosted GitLab support from day one (the user's own daily driver is a self-hosted work instance).

## What Changes

- **A third `PinNode` kind: `smart`.** A smart folder persists **config only** — `source` (`'gitlab'`, the only v1 value), instance `baseUrl` (self-hosted-critical; defaults to `https://gitlab.com`; validated as an absolute http(s) URL and stored without a trailing slash), a canned `query` enum (`authored | assigned | review-requested` — deliberately NOT a free-form query language), `refreshMinutes` (default 10, floor 5), and `name`/`icon` (minted as `folder-git-2`, a lucide glyph this change adds to the curated `ICON_NAMES` set). It lives in `pinnedBySpace` alongside tab/folder nodes, reorders among pins like a normal folder, and expands/collapses like one. This is the change's one persisted-schema migration: `CURRENT_SCHEMA_VERSION` 1 → 2 with an append-only pass-through migration (v1 data contains no smart nodes; the v2 `PinNode` union simply admits the kind).
- **Ephemeral results, never persisted.** Query results live in a new broadcast-only `AppState` slice (`smartFolders`, keyed by folder id: items + fetch state + fetchedAt), exactly the `liveTabsById` pattern — stripped in `persist()`, rebuilt by polls. After a SW restart a smart folder shows a quiet pending state until its first fetch lands.
- **Items are link-shaped, not tab-shaped.** No `SavedTab` records, no bindings, no drift, no boundary, no rename, no ✕. Activating a row dispatches the **existing** `openUrl` command into the active Space. New bus commands cover only the folder lifecycle: `createSmartFolder`, `updateSmartFolder`, `deleteSmartFolder`, `refreshSmartFolder` — update/delete/refresh address `{ spaceId, folderId }`, matching the existing folder-command convention.
- **An alarms-driven GitLab connector in the background** (the auto-archive precedent): one repeating alarm at the minimum configured cadence; each folder refreshes when its own `refreshMinutes` clock is due; plus refresh on sidebar open (a parallel `'lunma/state-request'` listener owned by the connector module — the pure-read snapshot handler is untouched) and a manual **Refresh now** entry in the folder's row menu. The connector *interface* is shaped against GitHub PRs / JIRA on paper (auth is a per-connector strategy because it does not generalize — GitHub's API is token-only), but **only the GitLab connector ships in code** — no stranded framework.
- **The agreed auth ladder (GitLab v1):** (1) a PAT, when configured, wins — `Bearer` header, read-only `read_api` scope, stored in `chrome.storage.local` (NEVER `storage.sync` — tokens must not ride a Google account across machines; this is why the PAT cannot use the sync-backed settings registry and gets its own `shared/connectors.ts` record); (2) otherwise **cookie-riding** — `fetch(api, { credentials: 'include' })` from the SW; the existing `host_permissions: <all_urls>` already exempts these requests from CORS and SameSite, so there is no manifest change, no `cookies` permission, and no secret stored — auth state mirrors the browser; (3) a cookie failure (401, 302-to-login, non-JSON body) renders a quiet "Sign in to GitLab" row that opens the instance; the next poll heals. Accepted, documented risk: session-cookie API access is a semi-documented GitLab contract.
- **Restraint-guarded presentation.** A result row is title + source icon + **one** status glyph — the MR's pipeline status as a small dot painted from the existing semantic tokens (`--success`/`--danger`/`--warning`/`--info`); approvals/unresolved threads live in the tooltip + ARIA label; draft-ness comes free in the title text. The folder header carries a quiet item-count badge. Failure and staleness are calm: last-known items hold in-session, offline/rate-limited shows a quiet inline note, never a red error card. A running pipeline is a static `--info` dot at every motion level (only the in-flight *refresh* indicator animates, and reduced motion holds it static). Result rows are activate-only (no drag/reorder of results; the folder node itself drags among pins like today's folders).
- **Creation + configuration UX.** A "New smart folder…" entry joins the existing "New folder" in the pinned-section header kebab (`RowMenu`), opening a config editor (source fixed to GitLab in v1; base URL, query enum, cadence). The options page gains a **Connectors** section for the per-instance PAT (a custom, `storage.local`-backed section — precedented by the shortcut-guidance card and the Recently-archived subpage, which also live outside the sync settings registry).

## Capabilities

### New Capabilities

- `smart-folders`: the smart-folder contract end to end — the persisted config node, the ephemeral results slice, polling/refresh scheduling, the GitLab connector with the auth ladder and its failure states, the row/folder presentation with the one-glyph restraint guard, the creation/config UX, and the options Connectors section.

### Modified Capabilities

- `storage-and-migrations`: `CURRENT_SCHEMA_VERSION` 1 → 2; the first real entry (`toVersion: 2`) appends to the empty baseline `migrations` list (a pass-through for v1 data); the persisted `PinNode` union gains the `smart` member; the validator becomes `AppStateV2Schema`. Every sibling requirement that pins `AppStateV1Schema`/`AppStateV1` by name (corruption quarantine, partial-corruption salvage — including its `salvagePersistedState(migrated): AppStateV2 | null` signature — ArchivedTab shape, the auto-archive override) is re-pointed to the v2 names in the same delta, so the post-archive spec stays self-consistent and a salvaged pinned tree containing smart nodes survives intact instead of being reset by v1 slice validation.
- `typed-message-bus`: four new commands (`createSmartFolder`, `updateSmartFolder`, `deleteSmartFolder`, `refreshSmartFolder`) with Zod payload schemas and handler error contracts; the `reorderPinned` `PinNode` payload schema mirror admits the `smart` kind; the stale standalone "reorderPinned command" requirement (still describing the pre-folder `{ spaceId; ids: SavedTabId[] }` payload) is rewritten to the current `{ spaceId; nodes: PinNode[] }` shape in the same delta, healing pre-existing spec drift; the `openUrl` requirement's dispatcher prose widens to include the sidebar's smart-folder result rows.
- `spaces-and-tabs`: the pinned-tree composition (shell + pinned list rendering and `reorderPinned` semantics) admits the smart node kind — a smart folder renders, expands, and reorders among pins like a folder, but its children are connector results, not pinned-tab rows. The drag requirement gains smart-node semantics: a smart row is an inert "onto" target (a drop on it lands as a nearest-edge top-level insertion), dragging a smart folder to the temporary section is rejected exactly like a folder, and no drop involving a smart node creates or fills a folder.
- `lunma-bookmark-bindings`: one MODIFIED block healing the third instance of the same pre-folder drift — the "Pinned ordering per Space" requirement still described `pinnedBySpace` as `{ [spaceId]: string[] }` with an ids-shaped `reorderPinned`; it is rewritten to the shipped `PinNode[]` tree (now including smart nodes) and the `{ spaceId, nodes }` payload.
- `chrome-event-coordination`: an ADDED requirement for the parallel `'lunma/state-request'` refresh-kick listener registered by `background/smart-folders.ts` — the pure-read snapshot-handler module is untouched. (The `PendingEvent` extension itself — the new `source: 'connector'` member and `smartFolders.result` kind — rides the `smart-folders` capability spec, per the living spec's extension rule for new sources/kinds.) Two MODIFIED blocks keep the living spec coherent: the pure-read channel's no-enqueue scenario is scoped to the snapshot handler (the kick listener may schedule work that enqueues on a later drain) and its snapshot-shape scenario re-points `AppStateV1Schema.parse` to v2; the "Sidebar-source handlers" requirement's stale `store.setPinned(spaceId, ids)` is healed to the shipped `(spaceId, nodes)` tree-replace.

- **Affected code**: `apps/extension/src/shared/` (`types.ts` — the `smart` PinNode variant + item/runtime types + the ephemeral `smartFolders` slice; `schemas.ts` + `migrations.ts` — the v2 bump; `chrome/storage.ts` — `readPersistedState`/`salvagePersistedState` re-pointed to `AppStateV2Schema`; `bus.ts` — four commands; `icon-names.ts` — `ICON_NAMES` gains `'folder-git-2'`), `apps/extension/src/ui/` (`icon-loaders.generated.ts` regenerated via `pnpm gen:icons`; `FolderRow.svelte` extended with an optional trailing badge, a menu-items override, a `busy` refresh-spin flag, and `panel`/`panelTitle`/`onPanelBack` pass-throughs to RowMenu's existing drill-in API), `apps/extension/src/background/` (new `smart-folders.ts` connector engine: alarm, due-folder scheduling, GitLab fetch/normalize/auth ladder, the parallel state-request refresh-kick listener; coordinator handlers + store mutators for folder lifecycle and runtime-slice writes; `index.ts` alarm wiring), `apps/extension/src/sidebar/` (smart-folder rendering composed by `PinnedTabs.svelte` + the config editor; the header-kebab "New smart folder…" entry; smart-node drag semantics in the drag controller), `apps/extension/src/options/Options.svelte` (the Connectors section), plus the matching vitest suites.
- **New public types/files/methods/fields** (exhaustive; anything beyond this is a deviation to raise first):
  - Types (`shared/types.ts`): the `{ kind: 'smart' }` `PinNode` variant; `SmartQuery` (`'authored' | 'assigned' | 'review-requested'`); `SmartFolderItem` (`{ id, title, url, status?: { tone, label } }`); `SmartFolderRuntime` (`{ state: 'pending' | 'ok' | 'signed-out' | 'error', items, fetchedAt }`); `AppState.smartFolders` (ephemeral, persist-stripped).
  - Schemas: the `smart` member on `PinNodeSchema` (persisted + the `bus.ts` mirror); `CURRENT_SCHEMA_VERSION = 2`; `AppStateV2Schema`; one `{ toVersion: 2 }` migration.
  - Bus: `createSmartFolder` (`{ spaceId, name, baseUrl, query, refreshMinutes }`), `updateSmartFolder` (`{ spaceId, folderId, name, baseUrl, query, refreshMinutes }`), `deleteSmartFolder` / `refreshSmartFolder` (`{ spaceId, folderId }`) commands + payload schemas.
  - Coordinator plumbing: the `PendingEvent` union gains the new source member `{ source: 'connector'; kind: 'smartFolders.result'; payload: { folderId, runtime } }` with matching handlers-map and `EventPolicy` entries; the repeating alarm is named `lunma/smart-folders-poll`; `background/smart-folders.ts` registers a parallel `'lunma/state-request'` refresh-kick listener (it never claims the message's response channel).
  - New files: `shared/connectors.ts` (the `lunma.connectors` `storage.local` record: per-host PATs via `readConnectors()` / `setConnectorToken(host, token | null)`), `background/smart-folders.ts` (connector engine), `sidebar/SmartFolder.svelte` (+ test harness/test), `sidebar/SmartFolderEditor.svelte` (+ test harness/test).

  - Icons: `ICON_NAMES` (`shared/icon-names.ts`) gains `'folder-git-2'`; `ui/icon-loaders.generated.ts` is regenerated to match. Accepted side effect: `ICON_NAMES` is the user-facing picker catalogue, so the glyph also becomes pickable for Space/folder identities — a reasonable work-context addition.
