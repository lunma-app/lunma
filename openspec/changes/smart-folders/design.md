# Design — smart-folders

## Context

The pinned tree (`pinnedBySpace: PinNode[]`) holds two node kinds today: `tab` (a `SavedTab` reference) and single-level `folder` (children = `SavedTabId`s). Both are user-curated. The sidebar renders the tree in `PinnedTabs.svelte`; folder lifecycle commands ride the typed bus; the persisted envelope sits at `CURRENT_SCHEMA_VERSION = 1` with an empty append-only `migrations` list (the clean pre-release baseline — this change adds the first real entry). The background already runs an alarms-driven periodic job (auto-archive: alarm → enqueue → coordinator drain → one persist + one broadcast), and `liveTabsById` establishes the ephemeral-slice pattern (`Persisted<T> = Omit<T, 'liveTabsById'>`, stripped in `persist()`). The manifest's `host_permissions: <all_urls>` exempts SW fetches from CORS and SameSite for any origin — the load-bearing fact that makes cookie-riding possible with zero manifest change. Settings are `chrome.storage.sync`-backed; secrets must not be.

Exploration with the user settled the v1 contract: GitLab MRs first (self-hosted work instance — base URL is v1-critical), cookie-default + PAT-override auth, pipeline status as the row's single glyph, ~10-min default cadence configurable per folder.

## Goals / Non-Goals

**Goals:**

- A smart folder in the pinned tree whose children are the user's live GitLab MRs, one click from any MR, with pipeline status at a glance.
- Zero-setup auth when the browser is already signed in to the instance; an explicit PAT override for hardened/short-session instances.
- Config persists; results never do. A restart costs one quiet pending beat, not stale or leaked data.
- The connector *interface* shaped so GitHub/JIRA connectors are additive later — without shipping any speculative framework now.
- Calm failure states: signed-out, offline, and rate-limited read as quiet notes, never red error cards.

**Non-Goals:**

- No GitHub/JIRA connectors in code (interface shaped on paper only; each is its own later change).
- No free-form query language — three canned queries (`authored | assigned | review-requested`).
- No notifications, no badge counts outside the folder, no "CI went green" alerting — this is a glanceable list, not a notification product.
- No OAuth flows (a later per-connector enhancement; the PAT field is the v1 escape hatch).
- No result-row drag, reorder, pin-to-keep, or rename — rows are activate-only links.
- No persisted result cache (privacy + simplicity; see D2).

## Decisions

**D1 — A third `PinNode` kind, not a parallel map.** The smart folder is `{ kind: 'smart', id, name, icon, source: 'gitlab', baseUrl, query, refreshMinutes }` inside `pinnedBySpace`. *Alternative:* a separate `smartFoldersBySpace` map — rejected: the folder must order among pins (drag a smart folder above a pinned tab), and the `PinNode[]` array IS the ordering; a parallel map would need its own interleaving index, a worse migration than widening the union. Children are NOT part of the node (no `children` field) — results are runtime data (D2). The node reuses the existing folder expand/collapse UI state mechanism (session-only, keyed by node id).

**D2 — Results are an ephemeral broadcast slice.** `AppState.smartFolders: { [folderId]: SmartFolderRuntime }` where runtime = `{ state: 'pending' | 'ok' | 'signed-out' | 'error', items: SmartFolderItem[], fetchedAt: number | null }`; `Persisted<T>` grows to omit it alongside `liveTabsById`, and `persist()` strips it. *Alternative:* persisting a warm cache for instant paint after a SW restart — rejected for v1: MR titles are work-sensitive; never persisting them is a real privacy property (nothing about your employer's MRs survives in `chrome.storage.local`), and the cost is one quiet pending beat per SW cold start. An item is `{ id, title, url, status?: { tone: 'ok' | 'pending' | 'warn' | 'fail', label } }` — the `tone` is semantic so the renderer maps it to tokens and a future GitHub connector reuses the shape unchanged.

**D3 — Single-writer discipline holds: fetches happen off-drain, results enqueue.** The connector engine performs network I/O OUTSIDE the coordinator drain (an async fetch must never block the event queue), then enqueues the internal event `{ source: 'connector', kind: 'smartFolders.result', payload: { folderId, runtime } }` — `'connector'` is a new `PendingEvent` `source` member (the chrome-event-coordination extension rule: new capabilities add sources/kinds in their own deltas, exactly how auto-archive introduced `'alarm'`), with matching handlers-map and `EventPolicy` entries in the same change (no coalescing — results are per-folder and infrequent); the coordinator's handler writes the slice via a store mutator (`setSmartFolderRuntime`) and the drain emits its one broadcast. *Alternative:* reusing `source: 'alarm'` — rejected: results also arrive from manual refresh and the sidebar-open kick, and diluting what `'alarm'` means trades honesty for one saved union member. This mirrors how auto-archive's alarm enqueues work rather than mutating directly. A refresh marks the folder `pending` (preserving last-known `items` so the list doesn't blink), fires the fetch via `ctx.runSideEffect`, and the result event lands on a later drain.

**D4 — Scheduling: one alarm, per-folder due-clocks.** A single repeating alarm (`lunma/smart-folders-poll`) at the minimum `refreshMinutes` across all smart folders (re-registered on folder create/update/delete; cleared when no smart folders exist — zero idle cost for non-users, exactly auto-archive's enable/disable pattern). Each alarm tick refreshes only folders whose `now - fetchedAt ≥ refreshMinutes`. Two freshness escape hatches: `background/smart-folders.ts` registers its own parallel `chrome.runtime.onMessage` listener for `'lunma/state-request'` (the sidebar's boot/open signal) that kicks the same refresh-due check — the pure-read snapshot handler in `state-snapshot-handler.ts` is NOT modified and its contract holds untouched (the parallel listener never calls `sendResponse`, never claims the response channel, and never blocks the snapshot); and the folder menu's **Refresh now** dispatches `refreshSmartFolder { spaceId, folderId }` (which refreshes unconditionally). `refreshMinutes` floor: 5 (rate-limit kindness); default 10.

**D5 — The auth ladder is per-request and stateless.** For a folder's instance host — derived as `new URL(baseUrl).host` (hostname plus any explicit port): (1) a PAT from `shared/connectors.ts` (the `lunma.connectors` record in `chrome.storage.local`, keyed by that host) → `Authorization: Bearer` and `credentials: 'omit'`; (2) no PAT → `credentials: 'include'` and the browser session rides along. Signed-out detection is response-shape-based, deliberately paranoid: a 401, a 3xx/`response.redirected` landing on HTML, or a non-JSON body all map to `state: 'signed-out'` — never an exception, never a red card. The token is never logged, never broadcast, never in `sync`. *Alternative:* OAuth (PKCE) — rejected for v1: real per-provider work, and self-hosted instances would each need an OAuth app registered; the PAT field covers the same ground for the users who need explicitness.

**D6 — GitLab REST, canned queries, bounded enrichment.** The connector calls documented v4 REST GETs (GraphQL is POST-only and session-cookie POSTs require a CSRF token — incompatible with cookie-riding, so REST it is): `GET {baseUrl}/api/v4/merge_requests?state=opened&per_page=20&...` with `scope=created_by_me` (authored), `scope=assigned_to_me` (assigned), or `scope=all&reviewer_id=<me>` (review-requested; `scope=all` is required — the endpoint defaults to `scope=created_by_me`, which would intersect the reviewer filter with your own authored MRs and return a near-empty review queue; "me" resolved once per poll via `GET /api/v4/user` and cached in-session). Results cap at 20 items (a review queue, not an archive; the folder badge shows `20+` when the page is full, so the cap is never silent). Pipeline status: the global list endpoint does not reliably carry `head_pipeline`, so the connector enriches with bounded per-MR detail fetches (concurrency ≤5, only for listed items); if implementation finds the list response already carries a usable pipeline field, the enrichment is skipped with no behavior change. An MR with no pipeline simply has no `status` (no glyph — absence is honest), and any pipeline status outside the mapped set (`manual`, `preparing`, `waiting_for_resource`, `scheduled`, or anything GitLab adds later) likewise maps to no `status` — absence over guessing.

**D7 — Calm failure presentation, last-known items hold.** `signed-out` → the folder shows one quiet row, "Sign in to ⟨host⟩", which `openUrl`s the instance; the next due poll heals. `error` (network/5xx/rate-limit) → last-known items stay rendered with a dim inline note row ("Couldn't reach ⟨host⟩") at the list end; `fetchedAt` drives nothing user-facing in v1 (no "updated 3m ago" timestamps — dashboard-speak). First-ever `pending` (no items yet) → three static ghost rows (low-alpha bars, no shimmer). Deleting a smart folder needs no two-step confirm: it destroys only its own config (recreatable in seconds), closes no tabs, and loses no user content — unlike `deleteSavedTab`.

**D8 — Rendering: a `SmartFolder.svelte` feature component composed by `PinnedTabs.svelte`.** The tree renderer branches on `node.kind === 'smart'` to this component, which composes the existing `ui/FolderRow.svelte` primitive — the same chrome `PinnedTabs` uses for `folder` nodes (row layout, chevron gutter, expand/collapse affordance, the `RowMenu` morph pattern) — extended in this change with: an optional trailing `badge`; a `menuItems` override (the built-in folder menu is folder-coupled — rename-in-place, IconPicker, two-step delete — wrong for a smart node; the override re-includes FolderRow's existing Move up/Move down pair, see below); a `busy` flag that spins the glyph during an in-flight refresh (the glyph lives inside FolderRow's internal markup, out of reach of feature-scoped CSS); and pass-through `panel`/`panelTitle`/`onPanelBack` props forwarded to `RowMenu`'s existing generic drill-in API (FolderRow merely hard-codes its own appearance panel today), so Edit… can drill in place (D9). The glyph tints with the active Space's colour (a smart node persists no colour of its own in v1). Result rows render feature-locally — they are NOT `TabRow`s (no close, no drag, no bound/drift semantics; impersonating `TabRow` would force every tab affordance to answer "what does this mean for an MR?"). *Alternative:* re-rolling the folder-row chrome inside `SmartFolder.svelte` — rejected: that duplicates a primitive, the exact thing the component-library policy forbids. The smart folder node itself drags/reorders among pins exactly like a `folder` node (full drag/drop semantics in D13); its menu (kebab + right-click) carries **Refresh now · Edit… · Move up · Move down · Delete** — Move up/down (disabled at the respective end of the top-level list, dispatching the full-tree `reorderPinned`) keeps the smart node's reorder keyboard-reachable like every other pinned row, per the living "Pinned rows reorder via keyboard-reachable menu actions" requirement, reusing the pair FolderRow's built-in menu already implements. Refresh is menu-only, no inline ↻ on the row (folder rows carry no inline action buttons; polling + the sidebar-open kick already cover freshness, so manual refresh is a rare escape hatch). Result rows are activate-only: click → `openUrl { url, windowId }`.

**D9 — Config editor as a menu drill-in panel, not a new surface.** "New smart folder…" joins "New folder" in the pinned-section header kebab; creation drills that kebab's `RowMenu` into a `SmartFolderEditor.svelte` panel (RowMenu's existing `panel`/`panelTitle` drill-in API — back-arrow header, fields in the panel), Edit… drills the smart row's own menu the same way via FolderRow's forwarded panel (D8) — the same in-place pattern the folder row's Icon & colour panel uses — and the right-click `ContextMenu` path drills into the same editor component (the `TabBoundaryEditor` precedent): instance base URL (a `TextInput`, default `https://gitlab.com`), query (a three-option `SegmentedControl`), refresh cadence (a `Select` over 5 / 10 / 30 / 60 min), name (a `TextInput`, auto-suggested from the query, e.g. "Review requests"). The base URL validates as an absolute http(s) URL (anything else blocks confirm with a quiet inline message; the SW re-validates and rejects with an error ack as the backstop) and is stored without a trailing slash; instances served under a subpath work unchanged, because every endpoint string-appends to `baseUrl`. Source is fixed (`gitlab`) in v1 and not shown as a choice. *Alternative:* a dedicated modal/morph à la `SpaceEditor` — rejected: heavier surface for four fields; the drill-in is already the pattern for in-place config.

**D10 — Connectors section in options, decoupled from folder configs.** The options page gains a custom **Connectors** card (outside the sync-backed settings registry — precedented by the shortcut-guidance card): it manages the `lunma.connectors` `storage.local` record directly — rows of `{ host, token-set indicator }` with add/replace/clear. It does NOT read folder configs (the options page has no `AppState` access today and shouldn't gain it for this); the editor (D9) hints "optionally add an access token in Settings → Connectors" and the signed-out row covers the token-less path. Token field is a password-type input; the stored value is never echoed back into the field (a "Token set — replace?" affordance instead).

**D11 — Schema v2: the first real migration.** `CURRENT_SCHEMA_VERSION` 1 → 2; `AppStateV2Schema` = V1 with the widened `PinNode` union; `migrations` gains its first entry `{ toVersion: 2, migrate: (raw) => raw }` — a pass-through, since v1 data cannot contain `smart` nodes and no field changes shape. The bump is deliberate even though old data parses under the new schema: it makes a DOWNGRADE detectable (an old build reading v2 data quarantines on the version gate instead of Zod-rejecting `smart` nodes with a confusing parse error). The `bus.ts` `PinNodeSchema` mirror gains the same member (`reorderPinned` must round-trip smart nodes losslessly).

**D12 — Coordinator handlers (lifecycle).** Payloads are flat (no new named config type) and address `{ spaceId, folderId }` like every existing folder command (`renameFolder`, `deleteFolder`, …): `createSmartFolder { spaceId, name, baseUrl, query, refreshMinutes }` mints the id (`crypto.randomUUID`) and the icon (the SW sets `icon: 'folder-git-2'` for the v1 source — a lucide glyph this change adds to the curated `ICON_NAMES` list; the node persists an `icon` field so a later change can expose it, but the v1 editor does not), normalizes `baseUrl` (absolute http(s) URL, trailing slash stripped — invalid throws, error ack), clamps the cadence, and calls `store.addSmartFolder(spaceId, node)`, which inserts the node at the **top** of the Space's pinned list (matching `createFolder`'s top insertion — both live in the same "New …" header menu) — the handler mints rather than the store because the immediate first fetch needs the new id, and mutators are void-returning; `updateSmartFolder { spaceId, folderId, name, baseUrl, query, refreshMinutes }` edits in place via `store.updateSmartFolder(spaceId, folderId, config)` (a `baseUrl`/`query` change invalidates `fetchedAt` → immediate refetch); `deleteSmartFolder { spaceId, folderId }` calls `store.deleteSmartFolder(spaceId, folderId)` (removes the node AND drops its runtime entry — both are store state) and retunes/clears the alarm; `refreshSmartFolder { spaceId, folderId }` refreshes unconditionally — it acks `ok` once the refresh is underway; the fetch *outcome* lands via the runtime slice, never the ack. Each create/update/delete also registers/retunes the poll alarm. Handler error contract: unknown `spaceId` or `folderId` throws (the ack carries the error); fetch failures never throw (they resolve to runtime states — D7). *Alternative:* `folderId`-only payloads (UUIDs are globally unique, the handler could search all Spaces) — rejected: diverges from every sibling folder command for no gain, and the O(1) `pinnedBySpace[spaceId]` lookup keeps the error contract identical to `deleteFolder`.

**D13 — Smart-node drag/drop: an inert "onto" target.** A smart node drags and reorders among pins as one unit, exactly like a `folder` node (it round-trips `reorderPinned` losslessly). But it is never an "onto" target: the drag controller SHALL NOT report a smart row via `targetOntoId` — a pinned or temporary tab dropped on a smart row lands as a nearest-edge top-level insertion (exactly like dropping between rows), never files "into" the smart folder (it has no `children`) and never mints a folder (folder-creation drops apply to pinned *tab* targets only). Dragging a smart folder into the temporary section is rejected like a folder — no state change, the node animates back. *Alternative:* a strict no-op for drops on smart rows — rejected: a dead drop zone mid-list reads as breakage during an ordinary reorder drag. Result rows are not drag sources or drop targets at all (D8).

**Spec-drift heals carried by this change** (commentary lives here, not in the delta blocks, which archive verbatim): the typed-message-bus "reorderPinned command" requirement, the chrome-event-coordination "Sidebar-source handlers …" requirement, and the lunma-bookmark-bindings "Pinned ordering per Space" requirement (`pinnedBySpace` as `{ [spaceId]: string[] }`) all still described the pre-folder ids shape / `store.setPinned(spaceId, ids)` — their MODIFIED blocks re-state the shipped `{ spaceId; nodes: PinNode[] }` tree-replace contract. Likewise, every living requirement that pinned `AppStateV1Schema`/`AppStateV1` by name (corruption quarantine, partial-corruption salvage — whose v1-pinned slice validation would otherwise have normatively *dropped* a salvaged pinned tree containing smart nodes — ArchivedTab shape, the auto-archive override, and the snapshot-shape scenario) is re-pointed to the v2 names so the post-archive specs stay self-consistent.

**Doc updates implied:** `docs/04-capabilities.md` — a new smart-folders capability section, the sidebar pinned-section composition bullet (smart node kind), and the options-page description (Connectors section). `docs/03-architecture.md` — the `PinNode` union prose gains the `smart` kind; the storage-schema section moves from the v1-empty-migrations baseline to v2 with the first real migration; the ephemeral-slice paragraph gains `smartFolders` alongside `liveTabsById`; and the section's stale `AppStateV5Schema`/`CURRENT_SCHEMA_VERSION = 5` remnants (pre-existing drift from the baseline reset) are corrected while in there. No other docs affected.

## Visual language

- **Hierarchy.** A smart folder reads as a folder first: same row height, leading icon (its configured icon — minted `folder-git-2` in v1, tinted with the active Space's colour like folder glyphs), name at title weight, the same expand chevron — plus a trailing quiet count badge (`--text-2xs`, `--text-faint`, a soft `--surface-2` pill; `20+` when capped). Children indent at the folder-children inset. The brand divergence from Arc is the point: Arc folders hold what you put in them; a Lunma smart folder is the Space *knowing what's waiting for you* — same furniture, live contents.
- **Result rows.** One line each: leading source favicon (`faviconFor(baseUrl)` — the instance's real icon), title in `--text-sm` sans ellipsized (`Draft:` prefixes arrive in the title for free), trailing **one** 8px status dot. Dot tones map to existing tokens — `ok → --success`, `fail → --danger`, `warn → --warning`, `pending → --info` — identical at every tint level (status is semantic, not Space-hued; a `gray` Space still shows a red failed pipeline). No text renders on the dot; the `Tooltip` + ARIA label carry the full phrase ("Pipeline failed · 2 approvals · 3 unresolved threads"), which is where approvals/threads live — the row itself never grows a second signal.
- **Interaction feedback.** Rows take the tab-row vocabulary: hover `--surface-2` wash, press `--press-scale`, the standard focus ring (`--focus-color`/`--focus-width`/`--focus-offset`), all over `--motion-fast` `--ease-standard`. The menu's **Refresh now** sets the folder icon spinning while in-flight at the established 0.8s-linear cadence (TabRow's loading spinner, reused verbatim by `FaviconTile`; feature-local token-referencing CSS, the same precedent); under reduced motion it holds static at a `--text-dim` tone instead of rotating — deliberately calmer than `FaviconTile`'s slowed-spin treatment, because a tab's loading progress is information while a refresh indicator is decoration.
- **States.** First-fetch pending: three static ghost rows (low-alpha `--surface` bars at row height — no shimmer, no strobe). Signed-out: one `--text-muted` row, "Sign in to ⟨host⟩", hover lifts to `--text`. Error: last-known rows unchanged + a dim `--text-faint` note row. Item-set changes swap quietly (`--motion-fast` opacity, no per-row entrance theatre; reduced motion = instant swap, identical end state).
- **Motion.** Expand/collapse inherits the existing folder behavior; nothing new animates beyond the in-flight spinner and the `--motion-fast` content swap. Reduced motion removes both with identical end states.
- **Contrast.** All row text uses existing text tokens over existing surfaces — no new fg/bg pair, so the automated contrast harness gains no new case; the status dot is a non-text glyph whose meaning is duplicated in the tooltip/ARIA text (WCAG 1.4.1: colour is not the only carrier).

## Risks / Trade-offs

- [Session-cookie API access is a semi-documented GitLab contract that upstream could harden] → The PAT override is first in the ladder and fully covers the path; the signed-out row makes degradation legible, not broken. Accepted knowingly in exploration.
- [Pipeline enrichment is N+1 (≤21 requests/folder/poll)] → Capped at 20 items, concurrency ≤5, 10-min default cadence — far inside GitLab's authenticated rate limits; the enrichment is skipped entirely if the list response already carries pipeline status.
- [Work-sensitive MR titles travel in the state broadcast to every surface] → In-memory only, never persisted (D2), never leaves the extension's own surfaces; accepted.
- [`review-requested` needs the user's id (`/api/v4/user`)] → Resolved once per poll cycle and cached in-session; a failure there degrades to the signed-out/error states like any other fetch.
- [Short enterprise session TTLs make cookie-riding nag] → Exactly what the per-host PAT is for; the editor hints at it.
- [A second smart folder on a different instance (work + gitlab.com)] → Already supported: `baseUrl` is per-folder, PATs are per-host; no extra design needed.
- [SW cold start shows pending instead of instant content] → One quiet beat at 3-ghost-rows; the deliberate price of never persisting results.

## Migration Plan

Schema v1 → v2: bump `CURRENT_SCHEMA_VERSION`, append the pass-through migration, rename/extend the current-version schema per the storage-and-migrations conventions. Pre-release: no released installs exist, so rollback concerns are limited to dev profiles (the quarantine fallback already covers a version-mismatch downgrade). No data transformation occurs.

## Implementation notes — agreed deviations (raised and approved during apply)

1. **`ui/TextInput` gained an optional `type?: 'text' | 'password'` prop.** The
   Connectors token field must be password-type (D10) and the primitive had no
   such option; extending the primitive beats re-rolling a raw input (the
   component-library policy).
2. **`sidebar/SectionHeader` gained `panel`/`panelTitle`/`onPanelBack` +
   bindable `open` pass-throughs** to its internal `RowMenu` — the plumbing D9's
   "New smart folder… drills the header kebab in place" requires (App closes the
   morph after the editor's confirm via the bindable `open`).
3. **Closing a menu with a forwarded drill-in active also fires `onPanelBack`**
   (FolderRow + SectionHeader), so the parent's editing flag resets and the next
   open lands on the action list, not a stale panel.
4. **The `smartFolders.result` handler drops a result that lands after its
   folder was deleted** — writing it would resurrect an orphan runtime entry the
   delete already cleaned up.
5. **`setSmartFolderRuntime` preserves last-known `items` on `error` too**
   (taking the attempt's `fetchedAt`): the calm-failure requirement's "errors
   keep last-known items" is implemented at the mutator, so the connector never
   needs to read the store off-drain.
6. **Pre-existing PinnedTabs bug fixed in passing** (exposed by the smart-node
   drag tests): `removeId` never removed a dragged folder node at top level, so
   the reinserted copy lost to the original under the SW sanitizer's
   first-occurrence-wins dedupe — a folder (now: any unit) drag-reorder DOWNWARD
   silently no-opped. `removeId` now removes any top-level node by exact id.
7. **Bounded fetch timeout** (dev-load finding: "eternal loading"). Connector
   requests carry `AbortSignal.timeout(20s)`; a hung connection (dropped VPN to
   a self-hosted instance) resolves to the quiet `error` state instead of
   wedging the folder in `pending` forever — the in-flight guard skips a folder
   while its fetch is outstanding, so an unbounded hang silently blocked every
   later poll. Spec delta's auth-ladder requirement amended.
8. **Post-boot refresh-due kick** (same finding). The runtime slice dies with
   the SW, so after every restart all folders are due; an ALREADY-OPEN sidebar
   never re-sends its `state-request`, so without this it stared at ghosts
   until the next alarm tick. `index.ts` runs `refreshDueSmartFolders` once
   post-boot. Spec delta's scheduling requirement amended.
9. **Sidebar holds last-shown items through reloads** (user-directed). The
   boot broadcast after an SW restart carries an empty `smartFolders` slice;
   `SmartFolder.svelte` now holds the last-shown item set in component memory
   (never persisted — D2's privacy property holds) and keeps it rendered and
   activatable while a reload is in flight; ghosts only before anything has
   ever been shown, and an `ok`-empty result clears the hold. Spec delta's
   calm-states requirement amended.
