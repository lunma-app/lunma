## Why

Users pick a lens-overview scope filter (repos/projects/feeds), open the overflow picker, and expect **Clear** to behave like every other multi-select Clear: deselect everything and show nothing for that axis. Today it doesn't — an empty filter is defined to mean "no filter, show everything," so Select all and Clear both persist the exact same stored state (`[]`) and produce the exact same visible result (everything shown). A user who clicks Clear expecting an empty board sees no change at all, which reads as a broken control. This change makes Clear actually filter to zero items, restoring the control's obvious, expected behavior, while preserving the existing "a fresh/never-touched filter auto-includes newly-arriving feeds/repos/projects" behavior for the genuinely-unfiltered case.

## What Changes

- **BREAKING** (spec-level, not a schema/migration break): redefine what `filter.<axis>` values mean for the lens-overview scope filters (`repos`, `projects`, `feeds` on `LensFilter`):
  - `undefined` (key absent) continues to mean **unfiltered** — every current and future-arriving facet value passes, matching today's "never filtered" behavior.
  - An explicit `[]` now means **filtered to nothing** — every row on that axis is excluded. This is new; today `[]` is treated identically to `undefined`.
- `applyLensFilter` (`shared/lens-filter.ts`) changes its empty-array handling: an axis with `array.length === 0` now excludes every row on that axis instead of imposing no constraint. `undefined` is unaffected (still no constraint).
- The lens-overview's `MultiSelect` scope pickers (Changes/Issues/Articles cards, `OverviewPage.svelte`) stop collapsing a full selection down to `[]`. Selecting every option (via "Select all" or by manually checking the last remaining option) now persists the explicit list of every currently-known value for that axis, so it is visually and semantically distinct from Clear's `[]`. This list still auto-widens as new facet values arrive in the *current* browsing session because the picker always offers the union of "present ∪ selected" — a value that arrives after the explicit list was written is simply an unchecked, filtered-out option until the user checks it (this is the one intentional behavior narrowing: selecting all no longer retroactively includes values discovered after the fact — see `design.md` for the tradeoff).
- `openspec/specs/lenses/spec.md` requirement text and scenarios (~lines 1656–1725, "Scope filters render inside their owning entity card" and its "Clicking Select all collapses to the unfiltered state" scenario) are rewritten to describe the new semantics.
- Out of scope: the separate **page-local, ephemeral** Articles-card feed filter (chips only, no persistence, described elsewhere in the same spec) is untouched — this change only affects the persisted `node.filter`-backed overview scope filters.

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `lenses`: the scope-filter requirement (repos/projects/feeds picker behavior) changes so an explicit empty selection (`[]`) means "matches nothing" rather than being a synonym for "unfiltered." The "Select all collapses to the unfiltered state" scenario is replaced with a scenario where Select all persists the explicit full value list, distinct from Clear's `[]`.
