## 1. `applyLensFilter` — explicit empty vs. absent

- [x] 1.1 In `apps/extension/src/shared/lens-filter.ts`, change `applyLensFilter`'s `hasEntities`/`hasRepos`/`hasProjects`/`hasFeeds` gates from `(array?.length ?? 0) > 0` to key-presence checks (`entities !== undefined`, etc.), so an explicit `[]` still builds an (empty) `Set` and the per-row check naturally excludes every row on that axis, while an absent key keeps building no `Set` (no constraint).
- [x] 1.2 Update the early-return (`if (!hasEntities && !hasRepos && !hasProjects && !hasFeeds) return rows;`) so it only short-circuits when every axis is absent — an explicit `[]` on any axis must fall through into the per-row filtering loop, not take the pass-through shortcut.
- [x] 1.3 Update `apps/extension/src/shared/lens-filter.test.ts`: add cases for an explicit `{ repos: [] }` / `{ projects: [] }` / `{ feeds: [] }` excluding every row on that axis while leaving other axes/entities untouched; keep/adjust the existing "empty filter passes everything" cases to use an absent key (`{}` or omitted) rather than an explicit empty array where the two now diverge.

## 2. `OverviewPage.svelte` — stop collapsing full selections to `[]`

- [x] 2.1 In `apps/extension/src/launcher/lenspage/OverviewPage.svelte`, change the three `MultiSelect` `onchange` handlers (repos/projects/feeds) to persist `vals` as-is via `setFilter`, removing the `vals.length >= visX.length ? [] : vals` collapse — a full selection (via Select all or manually checking every option) now persists the explicit list of every currently-known value.
- [x] 2.2 Update `filteredTagged`'s early-return guard (`!filter.entities?.length && !filter.repos?.length && !filter.projects?.length && !filter.feeds?.length`) to check key-presence (`filter.entities === undefined && ...`) instead of length, so an explicit `[]` on any axis flows into `applyLensFilter` (and is excluded there) rather than taking the "return everything" shortcut.
- [x] 2.3 Confirm the already-shipped ephemeral display state (`reposDisplay`/`projectsDisplay`/`feedsDisplay`, `lastNodeId` resync effect) needs no changes: it already defaults to "every visible option checked" when `filter.<axis>` is absent/empty-length, and mirrors whatever `vals` the user's last action produced — verify this still reads correctly now that Select all's stored value is the explicit list rather than `[]` (the popover's checked state comes from local `$state`, not from re-deriving off the stored value, so it should already be correct — this task is verification, not new code, unless testing surfaces a gap).

  **Gap found and fixed**: the lens-switch resync (`reposDisplay`/`projectsDisplay`/`feedsDisplay` initializer and `$effect`) used `filter.<axis>?.length ? filter.<axis> : visX`, which treats an explicit `[]` (now Clear's meaningful "excluded" state) identically to an absent axis, resyncing to "everything checked" instead of "nothing checked" when switching away from and back to a cleared lens. Changed to `filter.<axis> ?? visX` (`??` only falls back on `undefined`, preserving an explicit `[]`) in both the initializer and the resync effect. Stale comment describing the old collapse-to-`[]` rationale also updated.

## 5. Additional gaps found during manual (browser) verification — not in the original task list

None of these were named in `proposal.md`/`design.md`/`tasks.md` 1–4; each was found by actually exercising the fix end-to-end (task 4.3) rather than through unit tests, and each was confirmed with the user before fixing per the deviations policy. All are covered by `pnpm verify` (section 4.2, re-run after each) and the manual re-verification in 4.3.

- [x] 5.1 **Persistence-layer collapse bug**: `apps/extension/src/shared/store.svelte.ts`'s `setLensFilter` — the *only* real path from `OverviewPage.svelte`'s Clear/Select-all to persisted state (`setFilter` → `LensPage.svelte`'s `setLensFilter` bus command → this method) — independently collapsed **any** filter where every axis had `.length === 0` back to "delete `node.filter` entirely," using the exact same `(array?.length ?? 0) > 0`-style check the whole proposal is about removing, at a call site none of the artifacts named (design.md's Risk #3 anticipated a "missing call site" in the abstract but not this one specifically). In real usage this meant Clear's `{ feeds: [] }` never survived persistence — silently preserving the bug end-to-end despite every task-listed call site (1.1–2.2) being correctly fixed. Fixed `isEmpty` to key-presence semantics (`filter.entities === undefined && filter.repos === undefined && ...`), matching Decision 1. Updated `apps/extension/src/shared/store.lenses.test.ts` (rewrote the two tests that asserted the old collapse; added a test asserting `{ entities: [], repos: [], projects: [] }` persists as a real constraint, not a clear) — `coordinator.lenses.test.ts`'s existing `{}`-only test needed no change.
- [x] 5.2 **Entity-card dead-end**: `OverviewPage.svelte`'s four entity-card visibility gates (`{#if changes.length > 0}` etc., pre-existing code, not introduced by this change) gate on the *post-filter* count. Once Clear can genuinely zero out an axis (the point of this change), the entity card — including the scope-filter picker living inside it — unmounted entirely, leaving no UI path back to undo Clear short of hand-editing persisted state. `specs/lenses/spec.md`'s own scenario text ("the Articles card render**s** zero Articles") already implied the card should stay put; the implementation didn't match. Fixed by adding an unfiltered `byEntityAll` bucket (from `tagged`, not `filteredTagged`) and gating each card's visibility on `byEntityAll.<entity>.length > 0` instead of the filtered count; `empty` (the whole-board "nothing waiting" note) now also reads `byEntityAll` so it doesn't fight with a card that's merely filtered to zero. Added/extended `OverviewPage.test.ts` coverage (the 3.3 end-to-end Clear test now also asserts the card and its trigger stay mounted).
- [x] 5.3 **Trigger-label bug**: `scopeLabel()`'s `filter.<axis> ?? []` collapsed the genuinely-unfiltered (absent) and explicit-`[]` (cleared) states to the same argument, so a cleared picker's trigger still read "All feeds" — actively misleading (implies unfiltered/show-everything when the true state is "matches nothing"). Fixed `scopeLabel` to take `string[] | undefined` directly and branch on `undefined` (→ "All …") vs `[]` (→ new "None selected" copy) vs populated. Added the `launcher_lensNoneSelected` message key to all 9 locale files (`apps/extension/messages/*.json`) and regenerated `src/shared/paraglide` via `pnpm gen:i18n`. Added `OverviewPage.test.ts` coverage for both label states.
- [x] 5.4 **Popover-clipping bug (pre-existing, exposed by 5.2)**: `ui/MultiSelect.svelte`'s dropdown popover used plain CSS `position: absolute` inside the entity `.card`, which has `overflow: hidden` (for its rounded corners). Once 5.2 let a card render at near-zero height, the popover overflowed the card's clipped bounds and was invisible when opened — a real, user-blocking regression the user caught live. This was a latent bug in the primitive itself (not a semantics gap): `MultiSelect`'s dropdown mode was the only `mode="dropdown"` consumer in the codebase (the sidebar's `LensEditor.svelte` uses `mode="inline"`, unaffected), so it had never been exercised inside a short container before. `ui/Select.svelte` has the identical latent pattern but is out of scope here (never used inside a clipping container today; flagged as a follow-up, not fixed in this change). Fixed by rewriting `MultiSelect.svelte`'s dropdown-mode popover to use `bits-ui`'s `Popover.Root`/`Trigger`/`Portal`/`Content` (the same portal-escape pattern already established by `Menu.svelte` and `BottomSheet.svelte` in this codebase) instead of a bare `position: absolute` div — the popover now portals to `document.body`, escaping any ancestor's `overflow: hidden`. Preserved the exact existing visual/DOM contract (trigger classes, `.popover` box styling — renamed to the collision-safe global `.ms-popover` since a `class` prop forwarded into a foreign bits-ui component can't carry Svelte's scoped-CSS hash, mirroring `Menu.svelte`'s `:global(.lunma-menu)`) and kept the component's own roving-tabindex keyboard model (arrow keys/Home/End) rather than adopting bits-ui's own menu semantics; delegated outside-click/Escape dismissal to bits-ui's defaults (`trapFocus={false}` so Tab still leaves naturally, matching a plain field rather than a focus-trapped dialog). Updated `ui/MultiSelect.test.ts` and the `MultiSelect`-querying parts of `OverviewPage.test.ts` to query `document.body` (the portal target) instead of the render `container` for popover-internal elements — the standard testing-library fix for portaled content; trigger-only queries are unaffected. No catalog story changes needed (`MultiSelect.stories.svelte`'s public API/usage is unchanged). Re-verified visually in the browser (screenshot: the popover now renders fully, un-clipped, below a zero-height Changes card) and via the full round-trip (Clear → picker stays open and reachable → Select all → filter restored).
