# Lunma Component Library — Accessibility Audit

**Date:** 2026-06-29
**Method:** Source-grounded review — every component's `.svelte` source verified line-by-line, contrast math computed from `@lunma/tokens`, overlay/menu/dialog behaviour traced through `bits-ui`, each finding cross-checked by an independent adversarial pass.
**Coverage:** 37 `src/ui` primitives + the dev catalog shell, both themes (dark & light), all Colour-intensity levels.
**WCAG version:** WCAG 2.2 (AA primary target, AAA noted).

---

## Summary

The library's accessibility foundations are sound. Roles, names, and states are correct across the primitives; overlays (Tooltip, BottomSheet, Menu) delegate focus management, keyboard, and ARIA to `bits-ui`; `prefers-reduced-motion` is honoured in 13 primitives plus the token layer; and the shipped WCAG-AA contrast contract passes (103 assertions, `src/ui/contrast.test.ts`).

**44 open issues** — **0 Critical, 1 High, 9 Medium, 34 Low.** The single High is a light-theme contrast failure confined to the dev catalog (shipped light surfaces are unaffected). The rest are enhancement-grade gaps with no Critical among them: programmatic equivalents for visually-conveyed state (`active`/`busy`/drift), a few unnamed groups/controls, disabled-option roving in custom selects, and light-theme polish. Each is small, local, and individually fixable.

The bulk of the components ship correct, complete accessibility — catalogued under *Verified accessible* below.

## Section 1 — Catalog shell

### CAT-01 — Active nav item not programmatically current [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA). Dev-only catalog shell — never ships to users.
**Finding:** The currently-viewed story is marked only by a CSS class on the wrapping `<li class="nav-item" class:active={selected?.id === entry.id}>` (`apps/extension/catalog/Catalog.svelte:65`), which paints a wash via `.nav-item.active :global(.row-button)` (`Catalog.svelte:192-195`). The `RowButton` it renders is a plain `<button type="button">` with no `aria-current`/`aria-selected`/`aria-pressed` and exposes no such prop (`apps/extension/src/ui/RowButton.svelte:4-12,20-30`), so assistive tech cannot tell which component is selected.
**Fix:** Add an `aria-current` passthrough to `RowButton` and set `aria-current="page"` on the active nav button only.

### CATALOG-NEW1 — Boolean playground control lacks a prop-tied accessible name [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA). Dev-only catalog shell.
**Finding:** The boolean playground control renders a `Chip` toggle whose only accessible name is its label `'true'`/`'false'` (`apps/extension/catalog/lib/Story.svelte:75-80`); `Chip`'s toggle branch is `<button … aria-pressed>` with no `aria-label` and exposes no `ariaLabel` prop (`apps/extension/src/ui/Chip.svelte:45-57,60-77`). The sibling select/number/text controls each pass `ariaLabel={prop}` to `Select`/`TextInput` (`Story.svelte:86,90,97`), so they are prop-named; the boolean is not. The adjacent `SettingText label={prop}` (`Story.svelte:73`) renders `<span>`s, not a `<label for>` (`apps/extension/src/ui/SettingText.svelte:17-22`), so it is visible but not programmatically associated — AT announces only "true/false, toggle button, pressed" with no prop context.
**Fix:** Add an `ariaLabel` passthrough to `Chip` and pass `ariaLabel={prop}` for the boolean control so it matches the labelled `Select`/`TextInput` controls.

## Section 2 — Atoms

### AVATAR-01 — Verdict ring states differentiated by color only [Medium]
**WCAG:** 1.4.1 Use of Color (A)
**Finding:** The three verdict rings are color-only box-shadow halos — `approved` → `--success` (green), `changes` → `--danger` (red), `pending` → `--text-dim` (dim grey) (Avatar.svelte:63-71). Hue is the sole differentiator; the disc carries no glyph, shape, or per-state text, and the `approved`/`changes` pair is the classic red/green confusion case. The verdict reaches the accessible name only when a consumer threads it into `title` (e.g. `title="Ada Kale — approved"`, Avatar.stories.svelte:49-51), which `aria-label={title}` (Avatar.svelte:31) passes through — but that is a screen-reader name, not a visual cue, so it does **not** satisfy 1.4.1 for sighted colorblind users. Within the component there is no non-color affordance; any visual mitigation lives in adjacent verdict text in the composing surface, which the component does not guarantee. The ring is a supplementary status cue and the consuming surfaces conventionally show the verdict in nearby text, so severity is Medium.
**Fix:** Add a non-color cue per ring state (small verdict glyph/overlay, or a distinct ring style/pattern per state), or guarantee the verdict appears in adjacent visible text wherever the disc is composed.

### COLORSWATCH-01 — Swatch exposes `aria-pressed` toggle semantics inside consumer `role="radiogroup"` containers [Medium]
**WCAG:** 1.3.1 Info and Relationships (AA); 4.1.2 Name, Role, Value (AA)
**Finding:** `ColorSwatch.svelte:24-32` renders a native `<button type="button">` with `aria-pressed={selected}` (line 31) and `aria-label={color}` (line 32). Both shipped consumers place these swatches inside an ARIA `role="radiogroup"` container — `SpaceEditor.svelte:330-347` and `FolderRow.svelte:270-274`. A `radiogroup` requires owned `role="radio"` children that expose `aria-checked`; a toggle button carrying `aria-pressed` is not a radio, so each declared group owns zero conforming members. AT therefore gets no radio single-select / position-in-set ("X of Y") semantics, and the swatch is announced as a pressed toggle rather than a checked radio. The gap is wider in FolderRow, whose `role="radiogroup"` also lacks the arrow-key roving a radiogroup expects — there is no `onkeydown` and every swatch is left at the default `tabindex=0` (`FolderRow.svelte:270-274`) — so it is a radiogroup with neither radio children nor radio keyboard navigation. SpaceEditor does code the roving arrow-key pattern (`SpaceEditor.svelte:219-228, 337, 343`) but still feeds it button/`aria-pressed` members.
**Fix:** Minimal and complete — change both container roles from `role="radiogroup"` to `role="group"` (`SpaceEditor.svelte:333`, `FolderRow.svelte:270`); a labelled group of toggle buttons exposing `aria-pressed` is a consistent, valid pattern, needs no keyboard changes, and leaves the atom (and its standalone catalog use) untouched. Alternatively adopt the full radio pattern — give the swatch `role="radio"` + `aria-checked={selected}` in place of `aria-pressed`, AND add to FolderRow the roving arrow-key navigation SpaceEditor already has; changing the atom to `role="radio"` alone is insufficient because FolderRow's radiogroup would still lack the required arrow-key navigation between members.

### DIFFSTAT-01 — Additions/deletions counts carry no explicit programmatic label [Low]
**WCAG:** 1.3.1 Info and Relationships (A, borderline / best-practice)
**Finding:** The counts render as real text — `+{additions}` and `−{deletions}` (U+2212 MINUS SIGN) numerals at `apps/extension/src/ui/Diffstat.svelte:25-26` — and the decorative proportional bar is hidden from AT via `aria-hidden="true"` on its `.bar` wrapper (`apps/extension/src/ui/Diffstat.svelte:28`), which also hides both inner segments. The only distinction between the two numbers, beyond colour (`.add` → `--success`, `.del` → `--danger`, lines 51-56), is the `+` / `−` glyph prefix. Those glyphs are present in the DOM text, so the information is technically determinable, but screen readers at default punctuation verbosity commonly suppress `+` and `−`, so a non-sighted user can hear "128 84" with no cue which value is additions versus deletions. The component exposes no `aria-label`, `role="img"`, or visually-hidden "additions"/"deletions" text.
**Fix:** Append visually-hidden "additions"/"deletions" text after each numeral, or wrap the `.nums` span in `role="img"` with an `aria-label` like "128 additions, 84 deletions" (omitting an absent side). Leave the bar `aria-hidden`.

### FAVICONTENT-01 — Loading state carries no busy indication for assistive tech [Low]
**WCAG:** 4.1.3 Status Messages (AA)
**Finding:** The `loading` branch swaps the favicon for a decorative spinner — `<Icon name="loader-circle">` is rendered with no `label`, so Icon emits `aria-hidden="true"` (`FaviconTile.svelte:120`; `Icon.svelte:98-106`) — and the button's accessible name is unchanged (`aria-label={returnable ? 'Return to ...' : title}`, `FaviconTile.svelte:113`). The root `<button>` sets no `aria-busy` and there is no live region, so a screen-reader user who focuses the tile while it loads hears the *identical* name as the resting state and gets no signal the tile is busy. The status is genuinely not exposed, but it is transient and the control stays named and operable, so this is Low.
**Fix:** Set `aria-busy="true"` on the `<button>` while `loading` is true.

### FAVICONTILE-NEW1 — Off-home drift state not exposed; `aria-label` sits on a roleless `<span>` [Low]
**WCAG:** 1.1.1 Non-text Content (A); 4.1.2 Name, Role, Value (A)
**Finding:** The drift indicator is an empty `<span class="drift-dot" ... aria-label="Off home">` with no `role` (`FaviconTile.svelte:125`). On a generic (`<span>`) role, `aria-label` is name-from-author-prohibited, so assistive tech does not expose "Off home" (axe flags `aria-prohibited-attr`). When the tile is returnable (drifted *and* `homeHost`) the off-home state still reaches SR through the button's `aria-label="Return to {homeHost}"` (`FaviconTile.svelte:113,97`); but when `drifted` is true with no `homeHost` (`returnable` is false — `FaviconTile.svelte:97`), the dot is the only off-home cue and its name is dropped, leaving SR users nothing.
**Fix:** Drop the prohibited `aria-label`, mark the dot `aria-hidden="true"`, and convey the off-home state in the button's accessible name for the non-returnable drift case.

### FAVICONTILE-NEW2 — Active (current) state conveyed by background colour only [Low]
**WCAG:** 4.1.2 Name, Role, Value (A); 1.4.1 Use of Color (A)
**Finding:** The `active` treatment fills the plate with `--space-c-soft` (`FaviconTile.svelte:186-189`) as its sole indicator. The button sets `class:active` and a `data-active` test hook (`FaviconTile.svelte:106,112`) but no `aria-current` / `aria-pressed` / `aria-selected`, and the accessible name is unchanged (`aria-label={title}`, line 113). Neither a screen-reader nor a colour-vision-impaired user can tell which favorite's bound tab is currently focused.
**Fix:** Set `aria-current="true"` (or `aria-pressed={isActive}`) on the button when `isActive`.

### ICONBUTTON-NEW1 — Icon-only button can ship with no accessible name [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** `ariaLabel` is optional (`ariaLabel?: string | undefined`, IconButton.svelte:21) and the rendered icon is decorative — IconButton passes no `label` to `Icon` (IconButton.svelte:61), so `Icon` emits `aria-hidden="true"` (Icon.svelte:97-107) and contributes nothing to the name. The native `title` attribute (IconButton.svelte:53) is also optional and acts as the only name fallback. When a consumer omits both `ariaLabel` and `title`, the `<button>` (IconButton.svelte:51-62) has no accessible name, leaving an icon-only control unannounced to assistive tech. The type contract permits this — nothing requires a name on this icon-only primitive.
**Fix:** Make `ariaLabel` required on this icon-only control, or fall back to `title` as the accessible name when `ariaLabel` is absent, or dev-warn when both are missing.

### KBD-01 — Pictographic key glyphs have no text-alternative escape hatch [Low]
**WCAG:** 1.1.1 Non-text Content (AA, borderline) — closest fit; the glyph is also exposed as a text node, so this is a comprehensibility/pronunciation gap rather than a fully missing name.
**Finding:** `Kbd.svelte:4-6` declares `interface Props { children: Snippet }` — the only prop is `children`. The `<kbd class="kbd">` at `Kbd.svelte:11` carries no `aria-label`, no `aria-hidden`, and does not spread rest props. Consumers render pictographic symbols through it (`⌥L`, `⏎`, `⇥`, `⌘`, `K` — see `Kbd.stories.svelte:10,32,36,38,42`), and those symbols (⌘ = U+2318 "Place of Interest Sign", ⌥ = U+2325, ⇥ = U+21E5, ⏎ = U+23CE) are pictographs standing in for key names. The component gives no mechanism to attach a spelled-out accessible name ("Option L", "Command", "Tab") or to hide a purely decorative glyph, so a standalone Kbd cannot have its announcement corrected when no surrounding text already names the key. This is Low: the glyph is a real text node in the accessibility tree (not `aria-hidden`), key-cap hints are supplementary, and in product usage they typically sit beside descriptive text.
**Fix:** Add an optional `ariaLabel?: string` to `Props` and apply it to the `<kbd>` (`<kbd class="kbd" aria-label={ariaLabel}>`). Prefer an explicit prop over a broad rest-prop spread so consumers cannot also override `class`/`style`. This lets a consumer supply "Option L" or aria-hide a decorative glyph.

### TOOLTIP-01 — Tooltip content element lacks `role="tooltip"` [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** The portaled message element `<div {...props} class="lunma-tooltip">` (`apps/extension/src/ui/Tooltip.svelte:35`) renders as a generic `div`. The bits-ui content prop builder spreads `id`, `data-state`, `data-disabled`, transition attrs, an `outline:none` style and the content marker attribute, but no `role` (bits-ui content `props` at `bits-ui/dist/bits/tooltip/tooltip.svelte.js:665-674`), and Lunma adds none. The label text is still exposed to assistive tech: the trigger receives `aria-describedby` pointing at the content id while open (`bits-ui/dist/bits/tooltip/tooltip.svelte.js:580`, forwarded through `apps/extension/src/ui/Tooltip.svelte:28-30`), so screen readers announce the text as the trigger's accessible description. The residual gap is only that the bubble does not expose the `tooltip` role the WAI-ARIA Tooltip pattern recommends — an enhancement, not a broken description link.
**Fix:** Add `role="tooltip"` to the content div at `apps/extension/src/ui/Tooltip.svelte:35`.

### TOOLTIP-NEW1 — Catalog story spreads trigger props onto a non-interactive `<span>` wrapping `<Button>` [Low]
**WCAG:** 2.4.3 Focus Order (A); 1.4.13 Content on Hover or Focus (AA)
**Finding:** The story passes the bits-ui trigger props to `<span {...props}><Button …></span>` (`apps/extension/catalog/stories/ui/Tooltip.stories.svelte:40,48,54,62`). `Button.svelte` exposes a fixed Props interface and forwards nothing onto its `<button>` (`apps/extension/src/ui/Button.svelte:4-17,36-47`), so the trigger props cannot land on the control and are placed on a wrapper span instead. The bits-ui trigger defaults `tabindex` to `0` (`bits-ui/dist/bits/tooltip/components/tooltip-trigger.svelte:21`), so the span becomes a focusable tab stop nested around the already-focusable button — a redundant stop, with `onfocus`/`aria-describedby` on the wrapper rather than the control. Tabbing to the inner button shows no tooltip because `focus` does not bubble to the span's `onfocus`. This is dev-only (the catalog does not ship), and it contradicts the component's own documented contract — spread onto "your interactive element, e.g. `<button {...props}>`" (`apps/extension/src/ui/Tooltip.svelte:15-18`) — which every shipped consumer follows (`apps/extension/src/ui/FaviconTile.svelte:103`, `apps/extension/src/ui/TabRow.svelte:143`, `apps/extension/src/sidebar/Lens.svelte:718`).
**Fix:** In the story, spread the trigger props onto a native focusable `<button>`, or extend `Button.svelte` to accept and forward the trigger props, so one focusable element carries `aria-describedby` and the focus/pointer handlers.

## Section 3 — Composite components

### ACF-02 — Required token field exposes no programmatic required state [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** When `requirement === 'required'` the token field renders the bare label `'Token'` (vs `'Add a token (optional)'` for the optional variant) at AccountConnectField.svelte:89, but carries no `aria-required`/`required` attribute; TextInput exposes no prop to emit one (TextInput.svelte:2-30 has no `required`/`aria-required`, and the `<input>` at lines 65-79 emits none). Submission is gated by `disabled={token.trim() === ''}` on Connect (AccountConnectField.svelte:100), so the constraint is enforced behaviourally. The required variant has no visual required marker either (no asterisk, no "(required)" suffix), so sighted and AT users are at parity — a parity-neutral enhancement, not a missing-information defect.
**Fix:** Add an optional `required` prop to TextInput that emits `aria-required="true"`, and pass it from AccountConnectField when `requirement === 'required'`.

### ACF-03 — Error text not programmatically associated with the input [Low]
**WCAG:** 1.3.1 Info and Relationships (AA)
**Finding:** In the error state the input is marked invalid via `invalid={error !== undefined}` (AccountConnectField.svelte:93), which sets `aria-invalid={true}` (TextInput.svelte:74), and the message renders inside a `role="alert"` live region (AccountConnectField.svelte:118-119 → InlineError.svelte:19), so the error is identified and announced when it appears — WCAG 3.3.1 is substantively met. The remaining gap is purely the programmatic relationship: there is no `aria-describedby` tying the message to the input, because InlineError emits no `id` (InlineError.svelte:9-16) and TextInput exposes no `aria-describedby` prop (TextInput.svelte:2-30). On a later re-focus of the input the error is therefore not re-announced as the field's description.
**Fix:** Give InlineError a stable `id`, add an `aria-describedby` prop to TextInput, and wire the two in AccountConnectField when `error` is set.

### BOTTOMSHEET-01 — Headerless sheet renders an unnamed modal dialog; no prop exists to name it [Medium]
**WCAG:** 4.1.2 Name, Role, Value (A)
**Finding:** The bits-ui `Dialog.Content` `child` props are spread onto `.bottom-sheet` (`BottomSheet.svelte:81`), carrying `role="dialog"`/`aria-modal`. The accessible name comes only from `Dialog.Title`, which renders solely when `title` is set (`BottomSheet.svelte:82-84`); bits-ui wires `aria-labelledby` only when that `Title` is registered. The headerless path is first-class — `title?: string | undefined` (`BottomSheet.svelte:13`) plus a dedicated headerless story (`BottomSheet.stories.svelte:56-62`) — yet `Props` (`BottomSheet.svelte:5-30`) exposes no `ariaLabel`/`aria-label`. A sheet opened without a `title` is therefore an unnamed `role="dialog"` modal, and consumers have no escape hatch to name it.
**Fix:** Add an `ariaLabel?: string` prop. When `title` is omitted, either render a visually-hidden `Dialog.Title` (reusing bits-ui's own `aria-labelledby` wiring) or place the explicit `aria-label` AFTER the `{...props}` spread on `.bottom-sheet` (`BottomSheet.svelte:81`) so it wins. Document that headerless consumers must supply it.

### ICONPICKER-NEW1 — Search status messages not announced [Low]
**WCAG:** 4.1.3 Status Messages (AA)
**Finding:** As the query changes, IconPicker swaps in dynamic status text — the empty-result message `No icons match "…".` (`apps/extension/src/ui/IconPicker.svelte:169`) and the truncation hint `Showing the first 48 — keep typing to narrow.` (`apps/extension/src/ui/IconPicker.svelte:207-208`). Both render in plain `<p>` elements with no `aria-live`/`role="status"`. On an empty search the whole radiogroup is replaced by the `<p>` (the `{:else}` branch, line 170), so a screen-reader user gets no announcement that the grid is now empty or capped while typing.
**Fix:** Render the empty-result and truncation messages inside a persistent polite live region (`role="status"` or `aria-live="polite"` on a wrapper that is always in the DOM) so result-state changes are announced as the user types.

### LENSROW-01 — Active "peek" state has no programmatic equivalent [Medium]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** The `active` prop is conveyed only by CSS — a hue wash plus an inset ring applied by `.lens-row.active` (`LensRow.svelte:133-136`). The row's toggle button exposes `aria-expanded` but no current/selected state (`LensRow.svelte:78-79`), and the `.lens-row` container carries no `aria-current`, so a screen-reader user cannot tell which lens overview is the active peek. `aria-pressed` is the wrong tool — the button's action is expand/collapse (it already owns `aria-expanded`), not selection. (Note: 1.4.1 Use of Color does not apply — the active row is distinguished from the rest by the presence of an inset ring/border at `LensRow.svelte:135`, not by hue alone; hover uses the wash without the ring at `LensRow.svelte:128-130`, so a colour-blind sighted user can still tell active apart.)
**Fix:** Reflect the active peek with `aria-current` on the row — set `aria-current="true"` (or `"page"`) on the `.lens-row` container (or the toggle button) when `active` is true.

### LENSROW-02 — In-flight refresh is not perceivable to assistive tech [Low]
**WCAG:** 4.1.3 Status Messages (AA)
**Finding:** When `busy` is true the leading glyph spins (`LensRow.svelte:201-203`), but that glyph lives inside `<span class="tile" ... aria-hidden="true">` (`LensRow.svelte:82`), so the only refresh cue is hidden from assistive tech. No `aria-busy`, status text, or live region accompanies it. (`aria-busy="true"` alone would not announce anything — it suppresses subtree-change announcements, so it does not satisfy this on its own.) Whether this warrants a change depends on whether the ambient refresh is meant to be announced.
**Fix:** If the refresh state should be perceivable, add an `sr-only` polite live-region text gated on `busy` (e.g. "Refreshing {name}…"). The spinner glyph is correctly decorative and should stay `aria-hidden`.

### LENSROW-NEW1 — Trailing count badge is announced as a bare, unlabeled number [Low]
**WCAG:** 1.3.1 Info and Relationships (AA)
**Finding:** The count badge renders as a standalone, non-`aria-hidden` text node (`LensRow.svelte:92-93`) inside `.trailing`, a sibling of — not inside — the toggle button, whose accessible name is only "Expand/Collapse {name}" (`LensRow.svelte:79`). A screen reader reads the row as "Expand {name}, button" then a context-free "4", with no unit or association to the row.
**Fix:** Give the count programmatic context — fold it into the toggle's `aria-label` (e.g. "{name}, 4 items") or pair the visible digits with an `sr-only` unit suffix.

### RESULTLIST-01 — Listbox has no accessible name and no prop to supply one [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** The container renders `role="listbox"` (`apps/extension/src/ui/ResultList.svelte:119`) with `id={listboxId}` (default `'launcher-results'`, `:118,:48`) but no `aria-label`/`aria-labelledby`, and `Props` (`:13-44`) exposes no prop to set one. In the primary pattern the component delegates option announcement to a separate combobox input: it reports the active option id via `onactivedescendant` (`:74-76`) so the surface mirrors `aria-activedescendant` onto its input, and each option is announced that way — so the listbox's own name is rarely voiced. Note the input's accessible name names the *input*, not the listbox; `aria-controls` does not transfer a name, so the listbox remains unnamed. This is the reason severity is Low rather than higher: options stay announced even with the listbox unnamed. A secondary concern keeps it from being purely cosmetic — the listbox div is itself focusable and handles keydown directly (`tabindex="-1"` + `onkeydown={handleKeydown}`, `:120,:122`) and the doc comment documents a mode where "the list root itself receives the event" (`:90-95`); the div sets **no** `aria-activedescendant` on itself, so if a surface focuses it directly, arrow navigation is both unnamed and unannounced.
**Fix:** Add an optional `ariaLabel` prop forwarded to the listbox as `aria-label` (e.g. "Search results"); leave it unset where the combobox input already drives the announcement. Confirm at the surfaces whether the listbox div is ever focused directly (see manual-verify).

### RESULTLIST-02 — URL is included in the option accessible name (and duplicated as its description) [Low]
**WCAG:** 1.3.1 Info and Relationships (advisory)
**Finding:** Each option is a `<button role="option">` (`apps/extension/src/ui/ResultRow.svelte:53,:59`) whose accessible name is computed from its text content. The visible `.url` span is plain, not `aria-hidden` (`:90`), so the full URL is read as part of the name after the title and source badge — and re-read on every arrow keypress in the activedescendant pattern. Additionally `title={url}` (`:65`) supplies the button's accessible *description* (the `title` attribute is not used for the name when content already names the element), so the URL can also be announced a second time in verbose screen-reader modes. The `.url` span is ellipsised (`:256-263`), so `title={url}` is also the visual hover tooltip for the truncated URL and should be preserved.
**Fix:** Wrap the visible `.url` span in `aria-hidden="true"` so the URL leaves the accessible name; **keep** `title={url}` (it is the hover tooltip and at most a verbose-only description). The option then announces title + source once.

### RESULTROW-NEW1 — Option row is a native tab stop with no tabindex control [Low]
**WCAG:** 2.4.3 Focus Order (A)
**Finding:** The row is a native `<button type="button">` carrying `role="option"` (ResultRow.svelte:53-59) with no `tabindex` set and no `tabindex` prop on `Props` (ResultRow.svelte:6-36), so every row keeps the button's implicit `tabindex="0"` and stays in the tab sequence. The component's own doc comment describes the combobox `aria-activedescendant` model, where the input holds DOM focus and points at the row's `id` when it is the roving selection (ResultRow.svelte:29-31, 58). Under that model options must not be individual tab stops, but the primitive exposes no way to set `tabindex="-1"`, so a consumer wiring the documented active-descendant pattern cannot keep focus on the input — Tab walks through every result button. Whether this manifests depends on how the launcher composes the rows (see manualVerify).
**Fix:** Expose a `tabindex` prop (or render options with `tabindex="-1"`) so the consumer can keep DOM focus on the combobox input and drive selection solely via `aria-activedescendant`.

### REVIEWERRAIL-01 — Overflow badge `+N` lacks descriptive context [Low]
**WCAG:** 1.3.1 Info and Relationships (A)
**Finding:** The overflow badge renders as bare text — `<span class="overflow" data-testid="reviewer-overflow">+{overflow}</span>` (ReviewerRail.svelte:59) — with no accessible name beyond its text node. It is in the accessibility tree and is announced (not hidden, not non-text, not an interactive widget), so a screen-reader user hears "+2" with no word tying the count to reviewers. The named, labelled reviewer discs that precede it (ReviewerRail.svelte:53-56) make this interpretable in surrounding context, so the gap is minor.
**Fix:** Give the badge an explicit accessible name. Do NOT simply add `aria-label` to the bare `<span>` — a span has the generic role, which does not reliably expose an author-supplied name. Instead pair the name with a naming-capable role (e.g. `role="img" aria-label={`${overflow} more reviewers`}`) or keep the visible "+N" and append visually-hidden text (e.g. a `.sr-only` span reading " more reviewers").

### REVIEWERRAIL-02 — Reviewer cluster has no collective group name [Low]
**WCAG:** 1.3.1 Info and Relationships (A)
**Finding:** The rail container `<span class="rail">` (ReviewerRail.svelte:48) and the disc wrapper `<span class="discs">` (ReviewerRail.svelte:52) carry no `role="group"` and no accessible name, so the cluster is not labelled as "reviewers" on its own. Per-element semantics are largely present: the lead verdict glyph is labelled (ReviewerRail.svelte:50) and each disc is individually labelled via the forwarded `title` (ReviewerRail.svelte:55 → Avatar.svelte:30-32); only the collective label is absent. List semantics (`role="list"`/`<ul>`) are not appropriate — the intentionally overlapped tint-ringed discs are not an enumerated list. This is a minor enhancement, not a hard failure.

### SCP-01 — Programmatically-clicked OPML file input is an unnamed operable tab stop [Medium]
**WCAG:** 4.1.2 Name, Role, Value (A)
**Finding:** The OPML `<input type="file" class="sr-only">` (ServiceConnectPicker.svelte:260-267) is hidden via the clip-rect `.sr-only` rule (lines 367-377), which keeps it focusable and in the accessibility tree. It carries no `aria-label` and no associated `<label>`, so it exposes no accessible name. **Fix:** Add `tabindex="-1"` and `aria-hidden="true"` to the input (lines 260-267) so the programmatically-clicked control leaves the tab order and a11y tree; alternatively give it `aria-label="Import OPML file"` if it must stay reachable.

### SCP-03 — OPML confirm step reveals with no status announcement or focus move [Medium]
**WCAG:** 4.1.3 Status Messages (AA)
**Finding:** When `onFilePicked` sets `pendingFeeds` (ServiceConnectPicker.svelte:160), the `.import-confirm` panel (lines 269-294) renders with the result text "Found {N} feed(s) — …" plus the target `Select` and Cancel/Import buttons. The "Found N feeds" parse result is therefore never announced, so a screen-reader user is not told the file parsed successfully nor that a required confirmation step has appeared, and must navigate to discover it.
**Fix:** Wrap the result text (or the whole panel) in `aria-live="polite"` / `role="status"`, or move focus into the panel on reveal (e.g. a `tabindex="-1"` heading or the first control).

### TABROW-01 — Active tab state not programmatically exposed [Medium]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** When `active` is true the row gets only a visual Space-wash/ring plus a heavier title weight (`class:active` TabRow.svelte:100; `.tab-row.active` styles 229-232, 464-466) and a `data-active` test attribute (TabRow.svelte:105). Neither focusable control — the favicon button (TabRow.svelte:142-150) nor the title button (TabRow.svelte:160-166) — carries `aria-current`, `aria-selected`, or `aria-pressed`. `data-active` is a test hook, not surfaced to assistive tech, so a screen-reader user tabbing the list cannot determine which row is the current tab. The component owns the `active` prop and exposes no ARIA pass-through, so a consumer cannot supply this state either.
**Fix:** Set `aria-current="true"` on the title button (the primary focus target) when `active` is true, so the current tab is programmatically determinable.

### TABROW-02 — Loading state carries no programmatic signal [Low]
**WCAG:** 4.1.2 Name, Role, Value (AA)
**Finding:** With `loading` true the favicon slot swaps to a spinner (TabRow.svelte:90-91) and the title dims, but no `aria-busy` reflects the busy state. This is an enhancement gap, not a defect: the spinner Icon is decorative (`name="loader-circle"` with no label → `aria-hidden` via Icon's no-label branch, Icon.svelte:97-107), and the tab name stays on both the favicon button (`aria-label={title}`, TabRow.svelte:148) and the title button (TabRow.svelte:164), so nothing is mis-announced. The only missing piece is a programmatic "busy" cue.
**Fix:** Set `aria-busy="true"` on the row (or favicon button) while `loading` is true; keep the spinner `aria-hidden`.

### TABROW-NEW1 — Redundant focusable favicon button duplicates the title button on non-drifted rows [Low]
**WCAG:** 2.4.3 Focus Order (A)
**Finding:** On a non-drifted row the favicon button (TabRow.svelte:142-150) resolves to `aria-label={title}` and `onclick={() => onclick?.()}` — identical accessible name and identical action to the title button (TabRow.svelte:160-166). Both are natural `<button>` tab stops (no `tabindex` override on either), so every non-drifted row presents two adjacent stops that announce the same "{title}, button" and do the same thing, with the favicon button having no distinct function. Across a long tab list this doubles keyboard/screen-reader traversal. This is an efficiency/redundancy concern rather than a name/role/value failure — both names are correctly exposed. (When `returnable`, the favicon button is distinct: its name becomes `Return to ${homeHost}`, so this applies only to non-drifted/non-returnable rows.)
**Fix:** When not `returnable`, drop the favicon button from the tab order (`tabindex="-1"`) or render the favicon as a non-button, leaving the title button as the row's single keyboard/SR target. Mouse reach is unaffected (the `.row-focus` overlay and the favicon click still work).

### TOAST-02 — Message-only toast cannot be paused or dismissed early by keyboard-only users [Low]
**WCAG:** 2.2.1 Timing Adjustable (A)
**Finding:** Auto-dismiss is interruptible — the countdown pauses on pointer-enter and on focus-within and resumes with the time that remained (`Toast.svelte:51-60`, `72-89`), Escape from within dismisses immediately (`Toast.svelte:90-98`), and the default window is 5000ms (`Toast.svelte:24`, `27`). The gap is narrow and specific to message-only toasts: when `actionLabel` is absent the toast renders no focusable descendant (`Toast.svelte:119-121`) and the container itself carries no `tabindex` (`Toast.svelte:107-117`), so `onfocusin` and `onkeydown` never fire under keyboard navigation. A keyboard-only sighted user therefore cannot focus the toast to trigger the focus-within pause, cannot extend the ~5s window, and cannot reach the Escape handler — only pointer hover can engage a non-actionable toast. Action toasts are unaffected: the Undo `Button` at `Toast.svelte:120` is a natural tab stop, so focusing it pauses the timer and makes Escape reachable.
**Fix:** Give message-only toasts a keyboard-reachable engage path so the existing focus-within pause and Escape handler work without a pointer — always render a focusable dismiss control, or make the toast container a tab stop with `tabindex="0"` (plus a visible focus style). Note `tabindex="-1"` alone is insufficient: it allows programmatic/click focus but does not place the element in the keyboard tab order, so a keyboard-only user still cannot reach it.

## Section 4 — Form components

### MS-03 — Trigger's `aria-label` suppresses the visible selection summary, so the collapsed value is not exposed to assistive tech [Medium]
**WCAG:** 4.1.2 Name, Role, Value (also 2.5.3 Label in Name)
**Finding:** The dropdown trigger sets `aria-label={ariaLabel}` (MultiSelect.svelte:318) while rendering the parent-computed selection summary as its visible content `<span class="value">{label}</span>` (MultiSelect.svelte:322) — e.g. "All feeds", a feed name, or "3 feeds" (prop doc, MultiSelect.svelte:37-39). Because `aria-label` wins over an element's text content in the accessible-name computation, the accessible name is only the field name ("Filter by feed"); the visible summary never reaches the accessibility tree. A screen-reader user on the collapsed trigger hears the field name and expanded state but not the current value, and a speech-input user cannot target the control by the visible "3 feeds" text. The selection is recoverable only by opening the listbox and traversing every option's `aria-selected` state — degraded relative to a native `<select>`, which announces its value while collapsed.
**Fix:** Expose both the field name and the value — drop the trigger `aria-label` in favour of `aria-labelledby` referencing an external visually-hidden field label plus the `.value` span, or fold the summary into the name (e.g. `aria-label={`${ariaLabel}: ${label}`}`). Keep the listbox's own `aria-label` (MultiSelect.svelte:251).

### MS-04 — Disabled options use the native `disabled` attribute, so arrow-key roving focus stalls at a disabled boundary [Low]
**WCAG:** 2.1.1 Keyboard
**Finding:** Option rows render as `<button … disabled={option.disabled}>` (MultiSelect.svelte:265), and `optionEls()` collects every `[role="option"]` including disabled rows (MultiSelect.svelte:138-141). `moveTo()` calls `.focus()` on the computed index (MultiSelect.svelte:156-160); a native-disabled button cannot take focus, so the call is a no-op and `document.activeElement` is unchanged. With the catalog's trailing disabled row (MultiSelect.stories.svelte:34), ArrowDown from the last enabled option targets the disabled row and focus stalls, and ArrowUp from the first option likewise targets the disabled last row and stalls — up/down wrap-around is broken. This is not a full keyboard block in the common layout: all enabled options stay reachable by arrowing within the enabled range, and in `inline` mode the Tab handler is gated on `mode === 'dropdown'` (MultiSelect.svelte:193) so native tabbing reaches every enabled option and skips the disabled one. It becomes access-blocking only in the uncommon config where a disabled option is first with nothing pre-selected: `focusOnOpen()` targets index 0 (MultiSelect.svelte:149-153), the `.focus()` no-ops so focus stays on the trigger, and each ArrowDown → `moveTo(0)` re-targets the same disabled row, so arrow keys cannot enter the list at all.
**Fix:** Have `moveTo()` and `focusOnOpen()` skip rows whose option is disabled (advance to the next non-disabled `[role="option"]`). Optionally adopt the ARIA APG approach of `aria-disabled="true"` instead of native `disabled` (keeping the `toggle()` early-return guard, MultiSelect.svelte:121) so the row stays focusable/announced while skipped by activation.

### SF-01 — Input mode lacks the `placeholder` accessible-name fallback that trigger mode has [Low]
**WCAG:** 4.1.2 / 1.3.1
**Finding:** In `input` mode the field wires `aria-label={ariaLabel}` with no fallback (SearchField.svelte:126), while `trigger` mode wires `aria-label={ariaLabel ?? placeholder}` (SearchField.svelte:107). When a consumer omits the optional `ariaLabel` prop — as the default catalog usage does (SearchField.stories.svelte:57, interactive preview :44-51) — the input's only accessible name is the native `placeholder` attribute (SearchField.svelte:123, default `'Search…'` at :61). The native placeholder participates in the accessible-name computation, so the field is not strictly nameless; however that path is exposed inconsistently across browser/AT combinations and the visible text disappears on first keystroke. This is a robustness/consistency gap rather than a missing label — the `ariaLabel` prop exists and is wired.
**Fix:** Mirror trigger mode on the input — `aria-label={ariaLabel ?? placeholder}` (SearchField.svelte:126) — so an `ariaLabel`-less field still carries an explicit, AT-reliable name. Optionally have the default catalog story (SearchField.stories.svelte:57) pass `ariaLabel` to model the expected usage.

### SEL-03 — Disabled options break arrow-key roving (native `disabled` can't take focus) [Low]
**WCAG:** 2.1.1 Keyboard (AA)
**Finding:** `optionEls()` collects every `[role="option"]`, including disabled ones (Select.svelte:59-61), and `moveTo` focuses the target index unconditionally via `els[...]?.focus()` (Select.svelte:75-79). Options are native `<button disabled={option.disabled}>` (Select.svelte:180), and a disabled button cannot receive DOM focus, so `.focus()` is a no-op and `document.activeElement` does not advance. With the shipped trailing disabled `Custom…` option (Select.stories.svelte:33), ArrowDown from the last enabled option and ArrowUp from the first both target the disabled entry and stall, so wrap-around is dead in both directions, and `End` (`moveTo(els.length - 1)`, Select.svelte:108-110) lands on nothing. A disabled option placed mid-list likewise halts arrow traversal at that row (Home/End become the only way across). Every *enabled* option remains reachable via linear arrows plus Home/End, so this is degraded navigation rather than a hard keyboard trap, but the roving model should skip disabled rows.
**Fix:** In `moveTo`/`focusSelectedSoon`, advance to the next non-disabled `[role="option"]`; or keep options in the roving set with `aria-disabled="true"` (instead of the native `disabled` attribute) and ignore their activation.

### SEL-04 — Listbox options wrapped in `<li>`, not owned directly by `role="listbox"` [Low]
**WCAG:** 1.3.1 Info and Relationships (AA), 4.1.2 Name, Role, Value (AA)
**Finding:** Each `<button role="option">` is wrapped in a bare `<li>` sitting between the `<ul role="listbox">` and the option (Select.svelte:171, 173-174). The intervening `<li>` keeps its implicit `listitem` role, so the listbox's owned children are list items rather than `option`s — a structure `role="listbox"` does not permit (its required owned roles are `option`/`group`). Assistive tech may miscount set size/position or warn on the invalid child role.
**Fix:** Put `role="presentation"` (or `role="none"`) on the `<li>`, or drop the `<li>` and make the option buttons direct children of the listbox `<ul>`.

### TI-01 — No `aria-describedby`/`errorId` to associate a field's error text with the invalid input [Low]
**WCAG:** 3.3.1 Error Identification (A)
**Finding:** `aria-invalid` is wired correctly — `aria-invalid={invalid ? true : undefined}` (TextInput.svelte:74) applies `aria-invalid="true"` only when `invalid` is true and omits the attribute otherwise. The remaining gap is programmatic association: the `<input>` binds only explicit props with no attribute spread (TextInput.svelte:65-79) and exposes no `aria-describedby`/`errorId` prop, so a consumer that sets `invalid` cannot link the field to the visible error text the surrounding form renders. AT announces the field as invalid but not why on focus. This is an enhancement, not a hard violation — the surrounding form can still satisfy 3.3.1 by announcing the error through a live region (the repo's InlineError ships `role="alert"`); `aria-describedby` would add the on-focus/programmatic association.
**Fix:** Add an optional `errorId`/`describedById` prop forwarded to `aria-describedby` on the `<input>`, so the rejecting form can tie its visible error text to the field.

### TI-03 — No `autocomplete` passthrough [Low]
**WCAG:** 1.3.5 Identify Input Purpose (AA)
**Finding:** The `<input>` forwards `type` and `inputmode` but exposes no `autocomplete` prop, and binds no spread attributes (TextInput.svelte:65-79). For fields collecting information about the user (account name → `username`; the password/token field, `type='password'` per TextInput.svelte:23-25) consumers cannot set `autocomplete="username"` / `"current-password"`, so the input purpose cannot be programmatically determined.
**Fix:** Expose a typed `autocomplete` prop forwarded to the `<input>`.

## Section 5 — Layout components

### IE-02 — InlineError exposes no `id` for `aria-describedby` association [Low]
**WCAG:** 1.3.1 Info and Relationships (A), 4.1.2 Name, Role, Value (A)
**Finding:** The rendered element is `<p class="inline-error" role="alert" data-testid={testid}>{message}</p>` (`InlineError.svelte:19`), and the only props are `message` and `testid` (`InlineError.svelte:9-16`). The `role="alert"` announces the message assertively when the `<p>` mounts under a consumer's `{#if error}`, so the error is identified and described in text — WCAG 3.3.1 is met. The gap is that there is no stable `id` to forward onto the `<p>`: `data-testid` is not a valid `aria-describedby` target, so a consuming field cannot programmatically bind the error to its input. Every consumer renders the error adjacent to the field with no association — e.g. AccountConnectField wires `invalid={error !== undefined}` on its `TextInput` (`AccountConnectField.svelte:93`) yet renders `<InlineError message={error} … />` with no shared id (`AccountConnectField.svelte:119`); ServiceConnectPicker is the same (`ServiceConnectPicker.svelte:255,297`). A screen-reader user who returns focus to the field after the initial announcement does not hear the error as part of the field's accessible description.
**Fix:** Add an optional `id` prop forwarded onto the `<p>` so consumers can wire `aria-describedby` + `aria-invalid` on the input: `<TextInput invalid aria-describedby="tok-err"> … <InlineError id="tok-err" message={error} />`. `role="alert"` already covers the on-inject announcement, so this is a persistent-association enhancement, not an announcement fix.

## Section 6 — Theme & contrast

### THEME-01 — Glass surfaces have no light-theme value, so light-theme foreground tokens on `.lunma-glass` fail contrast [High]
**WCAG:** 1.4.3 Contrast Minimum (AA), 1.4.11 Non-text Contrast (AA)
**Scope:** Manifests on the dev-only component catalog. Shipped surfaces are unaffected (see below) — but the root cause is a shared design-token gap, so any future light-theme surface that uses glass inherits it.
**Finding:** `--glass-bg` is defined once, in the dark `:root` — `oklch(0.24 0.03 var(--base-hue) / 0.5)` (`packages/tokens/tokens.css:343`) — and is never overridden in the `[data-theme='light']` block. The shared `.lunma-glass` recipe paints `background: var(--glass-bg)` (`packages/tokens/recipes.css:118-125`), so in light theme a glass panel stays a dark, ~0.56-lightness brown while the rest of the frame is light. The catalog renders its component previews on glass — `.preview.lunma-glass` (`apps/extension/catalog/lib/Story.svelte:73`) and every example tile `.variant.lunma-glass` (`apps/extension/catalog/lib/Variant.svelte:13`) — and labels them with light-theme foreground tokens, which then sit on that dark panel:
- caption text `--text-muted` (figcaption, `Variant.svelte:46-49`) → **2.03:1**
- `--text-dim` → **1.54:1**
- block headings `--text-faint` (`Story.svelte` `.block-heading`) → **1.30:1**
- body `--text` → **4.77:1** (borderline)

All but body `--text` fail the 4.5:1 normal-text minimum; the same tokens clear 4.6–17:1 on the intended light `--surface`. The token contrast contract does not catch this — `contrast.test.ts` asserts text-on-glass only for the **dark** theme `--text`/`--text-2` over the aurora (`apps/extension/src/ui/contrast.test.ts:258-398`); the light-theme-on-glass pairing is never gated.
**Why shipped surfaces are safe:** the sidebar, new-tab, and launcher overlay each re-declare `--glass-bg` only under `:root:not([data-theme='light'])` (`apps/extension/src/sidebar/app.css:155,165`; `apps/extension/src/launcher/newtab/newtab.css:85,89`; `apps/extension/src/launcher/overlay.css:44`) and fall back to the opaque warm-paper `--atm-bg` panel in light theme, so they never place text on dark glass.
**Fix:** Either give `.lunma-glass` a light-theme `--glass-bg` (a light translucent fill under `[data-theme='light']`) so glass follows the theme, or have the catalog preview/example tiles use an opaque light surface in light theme (or keep previewing on the immersive dark substrate without flipping the foreground tokens to light). Then extend `contrast.test.ts` to gate light-theme foreground tokens on the composited glass surface so it cannot regress.

### THEME-02 — Idle TextInput/Select boundary falls below 3:1 in both themes [Medium]
**WCAG:** 1.4.11 Non-text Contrast (AA)
**Finding:** An idle `TextInput` is a recessed `--input` fill (`TextInput.svelte:110`) inside a 1px `--border` hairline (`TextInput.svelte:108`); `Select`'s trigger is the same pattern — `--bg` fill (`Select.svelte:234`) inside a 1px `--border` (`Select.svelte:232`). The fill provides essentially no boundary: `--input` vs `--surface` measures 1.06:1 dark / 1.07:1 light (`tokens.css:123`/`441` against `116`/`436`), so the border is the sole resting boundary cue. That border resolves to 1.37:1 on `--surface` and 1.46:1 on `--bg` in dark (`--border` `oklch(0.300 …)`, `tokens.css:125`), and 1.45:1 / 1.32:1 in light (`--border` `oklch(0.875 …)`, `tokens.css:443`) — all under the 3:1 required to identify a component boundary. The hover step `--border-strong` only reaches 1.86–1.99:1 (`tokens.css:127`/`445`), still under 3:1; focus is fine (accent border + `--bg-elev` fill, `TextInput.svelte:129-131`, `Select.svelte:250-252`), so the gap is the resting/unfocused state. `contrast.test.ts` asserts only text-on-bg pairs (`BGS`, line 124) — no border-vs-surface 3:1 assertion exists, so this pair is ungated.
**Fix:** Darken `--border`/`--border-strong` in both the dark `:root` and the `[data-theme='light']` block so an idle input/select boundary reaches ≥3:1 against `--surface` and `--bg`, or give idle fields a fill that differs ≥3:1 from the surface. Add a non-text 3:1 assertion for these border-vs-surface pairs to `contrast.test.ts`.

### THEME-NEW1 — `--text-faint` undershoots 4.5:1 where it carries meaningful normal-size metadata [Low]
**WCAG:** 1.4.3 Contrast Minimum (AA)
**Finding:** `--text-faint` is contract-gated only at AA Large 3:1 (`contrast.test.ts:162-173`), and `readTokens()` parses only the dark `:root` block (`contrast.test.ts:46-49`), so the light-theme value is ungated entirely. Light `--text-faint` `oklch(0.560 …)` (`tokens.css:453`) measures 4.21:1 on `--bg`, 3.97:1 on `--surface-2`, 3.51:1 on `--surface-3`; dark `--text-faint` `oklch(0.600 …)` (`tokens.css:138`) measures 4.22:1 on `--surface-3` — all below the 4.5:1 normal-text threshold. It is applied to informative, normal-size copy, not only decorative/disabled text: `TabRow.svelte:499` renders the at-rest `.meta` (an archived age + delete countdown) at `--text-xs` (11px, `tokens.css:262`), and `Menu.svelte:257` renders the section-kind label at `--text-2xs` (10px). 11px/10px regular is WCAG normal text (needs 4.5:1), so on the lighter surfaces this meaningful metadata is sub-4.5:1.
**Fix:** Raise `--text-faint` lightness so it clears 4.5:1 on every surface in both themes, or formally restrict `--text-faint` to genuinely incidental/decorative text (placeholders, disabled rows, glyphs) and move informative metadata to `--text-dim`; then extend `contrast.test.ts` to gate the light-theme block too.

## Section 7 — API consistency

### API-03 — `label` prop carries two different meanings across primitives [Low]
**WCAG:** — (API consistency; no AT failure)
**Finding:** The `label` prop names *visible text* on some primitives and an *accessible-name override* on others. `TextInput` renders `label` as a visible `<span class="label">` (`TextInput.svelte:61-63`) and `RowButton` renders it as the button's visible text (`RowButton.svelte:29`). By contrast `Menu` uses `label` purely as the trigger/popover accessible name with no visible text (`Menu.svelte:31-32,63`), and both `FolderRow` and `LensRow` use `label` as an `aria-label` override while a separate `name` prop carries the visible text (`FolderRow.svelte:31-32,223`; `LensRow.svelte:34-35,79`). `Button` exposes no `label` prop — its content comes from `children` (`Button.svelte:16`). Every usage produces a correct accessible name with a safe default, so there is no AT failure; the cost is API predictability.
**Fix:** Document the convention (`label` = visible text that doubles as the accessible name; `ariaLabel` = name only) and rename the accessible-name-only `label` on `Menu`, `FolderRow`, and `LensRow` to `ariaLabel`.

### API-04 — `TextInput` exposes no `id` / `aria-describedby` passthrough for external associations [Low]
**WCAG:** — (enhancement; adjacent to 3.3.1 Error Identification)
**Finding:** `TextInput`'s props (`TextInput.svelte:2-30`) include no `id`, `aria-describedby`, or `aria-errormessage` passthrough, so a consumer cannot programmatically associate an external description or error message with the field. `AccountConnectField` composes `TextInput` (`AccountConnectField.svelte:87`) with a sibling `InlineError` (`AccountConnectField.svelte:119`) that is consequently unlinked to the input. Impact is limited: the field is self-labelled via its wrapping `<label>` / `aria-label` (`TextInput.svelte:60,73`), `invalid` already drives `aria-invalid` (`TextInput.svelte:74`), and the error announces on render through `InlineError`'s `role="alert"` (`InlineError.svelte:19`). The residual gap is that re-focusing an already-errored field does not re-announce the message, because nothing links it to the field.
**Fix:** Add an optional `id` (and/or `describedById`) passthrough to `TextInput` so a paired `InlineError` / help line can be linked via `aria-describedby` / `aria-errormessage`.

### API-06 — Busy/loading state is not programmatically exposed [Low]
**WCAG:** 4.1.3 Status Messages (AA) — borderline
**Finding:** `FolderRow`, `LensRow`, `FaviconTile`, and `TabRow` accept a `busy`/`loading` prop that animates a spinner, but none set `aria-busy` or surface the state through any role/property — no `aria-busy` appears anywhere in `src/ui`. The spinners are decorative and correctly silent: FolderRow's and LensRow's spinning glyph wrappers are `aria-hidden` (`FolderRow.svelte:208,230`; `LensRow.svelte:82`), and FaviconTile's/TabRow's `loader-circle` spinner is an unlabelled `<Icon>`, which renders `aria-hidden` (`Icon.svelte:97-107`; `FaviconTile.svelte:120`; `TabRow.svelte:91`). Each control keeps its accessible name throughout (`FolderRow.svelte:223`; `LensRow.svelte:79`; `FaviconTile.svelte:113`; `TabRow.svelte:148,165`). Nothing breaks — an AT user simply gets no signal that a refresh/load is in flight.
**Fix:** Where the load is user-meaningful, set `aria-busy="true"` on the row/tile while `busy`/`loading` (FolderRow, LensRow, TabRow). For the deliberately decorative favicon refresh, leaving it silent is acceptable — document that choice.
