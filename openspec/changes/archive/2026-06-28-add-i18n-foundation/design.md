## Context

Lunma has no i18n: every UI string is hardcoded English and the manifest (hence the
Chrome Web Store listing) is English-only. This change introduces the i18n foundation —
library, message catalog, locale resolver, the `language` preference, and native
manifest localization — plus one user-visible slice (the Options language picker and a
localized install listing). The broad migration of the ~124 hardcoded surface strings is
the named downstream change `localize-extension-ui`; the marketing site is
`localize-marketing-site` (a separate inlang project — cross-app imports are
Biome-banned).

Constraints that shape the design:
- **Pinned stack** (`docs/tech-stack.md`): adding a library is a stack change requiring
  this proposal + a docs update in the same change.
- **Import DAG** (Biome `noRestrictedImports` + `noImportCycles`): `shared/` may import
  nothing else in `apps/extension/src/`; `settings.ts` importing the i18n message
  catalog would risk a `settings ↔ i18n` cycle.
- **Service worker**: the `shared/` bundle is loaded in the SW, which has no
  `window`/`document`/`localStorage`.
- **Async storage, sync runtime**: the locale lives in `chrome.storage.sync` (async),
  but Paraglide's `getLocale()` is synchronous.
- **Boot seams differ per surface**: `sidebar/main.ts` and `launcher/newtab/main.ts`
  already run an async `boot()` that `await readSettings()` *before* `mount()` (the
  natural place to seed the locale). `options/main.ts`, however, mounts **synchronously**
  (7 lines) and `Options.svelte` reads settings in `onMount` — *after* first paint — so
  seeding the Options locale before paint requires restructuring `options/main.ts` into an
  async boot (new, surface-local boot machinery for that one surface).
- **Live-apply, not reload, is today's pattern**: every surface's `watchSettings`
  callback applies density/theme/tint/glares **in place** with no reload; no surface calls
  `location.reload()`. Options has **no** `watchSettings` subscription at all. A
  language switch therefore *adds* new wiring rather than reusing an existing reload path.
- **Generated-code precedent**: `src/ui/icon-loaders.generated.ts` is committed and
  regenerated by a `gen:*` script prepended to `dev`/`build`.

## Goals / Non-Goals

**Goals:**
- Pick and pin a compiler-based i18n library that gives type-safe, tree-shakeable
  messages, ICU plurals, and runtime locale switching.
- A locale resolver that is SW-safe, backed by the Settings store, and seeds before
  first paint.
- A user-selectable `language` setting (default `auto` = browser locale) rendered with
  the existing Options controls.
- Native `_locales` manifest localization for the store listing.
- A parity test guaranteeing catalog completeness.
- Keep `pnpm verify` green; keep the import DAG legal.

**Non-Goals:**
- Migrating the existing hardcoded surface strings (→ `localize-extension-ui`).
- Authoring real translations (the 8 non-`en` catalogs ship as English copies here; the
  consumer change fills them with context-aware, length-conscious translations).
- Marketing-site localization (→ `localize-marketing-site`).
- Per-locale fonts or RTL (all 9 target locales are LTR; CJK falls back to a system
  face, addressed in the consumer/site changes).
- A bespoke language-picker primitive (the existing `Select` path suffices).

## Decisions

### D1 — Library: `@inlang/paraglide-js` (v2)
Compiler emits tree-shakeable, type-safe message functions with a tiny runtime, native
ICU plurals, and runtime locale switching via a pluggable strategy system.
- **Alternatives:** `svelte-i18n` (heavier runtime store, weaker types); `typesafe-i18n`
  (maintenance concerns); **native `chrome.i18n` only** (rejected for in-app UI — it
  follows the browser UI language with no runtime switch and weak plural/ICU support, so
  it can't satisfy the user-selectable requirement; it *is* used for the manifest, where
  it's the only option — see D5).
- **Doc impact:** add Paraglide to `docs/tech-stack.md` pinned stack.

### D2 — Generated runtime in `shared/`, committed
Compile target `apps/extension/src/shared/paraglide/`. In `shared/` it is DAG-legal
(imports nothing from other layers; every surface may use it). Committed + regenerated by
`gen:i18n` (prepended to `dev`/`build`, like `gen:icons`) so standalone `tsc`/
`svelte-check` (which run outside Vite) see the `.d.ts`.
- **Alternative:** gitignore the output and prepend `gen:i18n` to `typecheck`/`check`/
  `lint`/`test`/`build`. Rejected: more script churn and a clean checkout fails
  `pnpm typecheck` without the prestep; committing matches existing precedent.
- **Biome:** add `"!apps/extension/src/shared/paraglide/**"` to `files.includes` so
  generated `.js` isn't linted.

### D3 — SW-safe custom locale strategy
Strategy array is `['custom-lunmaSettings', 'baseLocale']` only. A
`defineCustomClientStrategy('custom-lunmaSettings', { getLocale, setLocale })` reads/writes
a module-level sync `cached` value. The
default `url`/`cookie`/`localStorage`/`preferredLanguage` strategies touch
`window`/`document`/`localStorage` and would throw in the SW — they are deliberately
omitted.
- **The strategy's `setLocale` updates the in-memory cache ONLY — it does NOT persist.**
  Paraglide's `getLocale()` self-heals on its first call by invoking
  `setLocale(resolved, { reload: false })` (the `localeInitiallySet` path); a persisting
  strategy would therefore writeSetting the resolved concrete locale over the stored
  `'auto'` sentinel on the very first `m.*` render — permanently de-selecting "System" and
  firing a spurious first-paint reload. Persistence is owned solely by the exported
  `setLocale` wrapper (the Options picker path), which `writeSetting('language', value)`
  for the raw choice (incl. `'auto'`). *(Corrected in `localize-extension-ui` once surface
  strings actually consumed `m.*` and activated the latent self-heal; the original
  foundation draft had the strategy persist.)*
- **Async bridge:** `getLocale()` is synchronous and returns the in-memory `cached`
  locale; the cache is filled asynchronously by `initLocale()` (which `await`s
  `readSettings()`) inside each surface's existing pre-`mount()` seam. Unseeded (jsdom
  tests) → `getLocale()` returns `undefined` → Paraglide falls through to `baseLocale`
  (`en`), which keeps existing English-asserting tests green.
- **Alternative:** `overwriteGetLocale`. Rejected: the strategy array gives a cleaner
  `custom → baseLocale` fallback chain and a single registration point.

### D4 — `language` setting with an `'auto'` sentinel; no settings→i18n edge
`settings.ts` gains `language: SupportedLocale | 'auto'`, `DEFAULTS.language = 'auto'`,
and an enum declaration whose options are `auto` (labelled "System") + 9 **endonyms**.
`'auto'` resolves to
the browser locale inside `initLocale()`, not in `DEFAULTS`, so the default stays static
and the choice keeps tracking the browser if the user later changes Chrome's UI language.
- **Cycle avoidance:** endonym labels ("Español", "日本語", …) are literal strings — a
  language name is shown in its own language and needs no translation. So `settings.ts`
  imports no message catalog and the `settings ↔ i18n` cycle never forms. (`i18n.ts`
  imports `settings.ts`, one-directional.)
- **Alternative:** make setting labels message-backed thunks. Rejected here — unnecessary
  for endonyms and would create the cycle. (The *other* settings' label localization is
  handled in `localize-extension-ui` via an `options/labels.ts` indirection, not in
  `settings.ts`.)

### D5 — Manifest via native `chrome.i18n` `_locales`
A library cannot localize the manifest, so `default_locale: "en"` + `__MSG_*__` +
`public/_locales/{locale}/messages.json` is the only mechanism. Localize `description`,
`action.default_title`, and the two command descriptions; leave brand `name`/`short_name`
literal. crxjs passes `public/` through verbatim.
- **Doc impact:** document `_locales` in `docs/architecture.md`.

### D6 — Reload on a *language* change only; everything else stays live-apply
No surface reloads today; `watchSettings` applies each setting in place. The language
switch adds reload wiring because Paraglide messages are resolved at evaluation time and
a reload is the simplest correct way to re-render every mounted tree in the new locale.
- Each surface's `watchSettings` callback SHALL **diff the previous vs new `language`**
  and call `applyLocaleFromSettings(language)` + `location.reload()` **only when
  `language` actually changed** — density/theme/tint/glares MUST keep applying live, with
  no reload, exactly as today. An unconditional reload here would be a regression (every
  density/tint toggle would reload the surface).
- `sidebar/main.ts` and `launcher/newtab/main.ts` already have a `watchSettings`
  subscription to extend with the gated language branch. `Options.svelte` has **none** —
  this change adds a new `watchSettings` subscription to Options for the reload path.
- `setLocale` is called with `{ reload: false }` so the gated `watchSettings` path is the
  single owner of the reload (no double reload on the initiating surface). The overlay,
  dormant between opens, picks up the locale on next open.
- **Alternative:** wrap `getLocale()` in a Svelte `$state` for flicker-free in-place
  re-render. Deferred as polish — heavier, not needed for a settings-driven switch.

### D8 — `SupportedLocale` derived from the generated locale set (no triple-source drift)
The supported-locale set exists in three places: `project.inlang/settings.json`
(`locales`), Paraglide's generated `locales`/`Locale` (in `shared/paraglide/`), and the
`SupportedLocale` type in `settings.ts`. To avoid drift, `SupportedLocale` SHALL be
derived from the generated runtime's `locales` constant (a `shared → shared` edge — the
generated runtime imports nothing from other layers, so there is no cycle and this is
*not* the catalog/endonym dependency D4 avoids). A unit test additionally asserts the
inlang `locales`, the generated `locales`, and `SupportedLocale` agree.
- **Alternative:** hand-write the union literal in `settings.ts`. Rejected — three
  independent sources silently drift; the parity test only checks catalog *keys*, not the
  locale *set*.

### D7 — Parity test reads catalogs at runtime
`i18n-parity.test.ts` uses `readFileSync` (not static `import`) so `messages/` and
`_locales/` stay outside the TS/Biome/Stylelint graph (mirrors `version-parity.test.ts`
reading `package.json`).

### Doc updates in this change
- `docs/tech-stack.md`: add `@inlang/paraglide-js` to the pinned stack with rationale.
- `docs/architecture.md`: document the `shared/paraglide` placement, the SW-safe custom
  strategy + resolver (`i18n.ts`), and native `_locales` manifest localization.
- Left untouched: all other `docs/`.

## Visual language

The only user-visible surface this change ships is the **Language** control on the
Options page. It composes existing primitives — no new primitive, no re-rolled control.

- **Control & hierarchy:** a `Select` dropdown (the enum has 10 options, exceeding the
  `≤4` `SegmentedControl` threshold), rendered by the existing declarative Options engine
  under the **Appearance** group, **stacked beneath its "Language" label** (the engine's
  wide-control rule) with a one-line description. It sits alongside Density / Colour
  intensity / Background effects, visually identical to the existing `defaultSearchEngine`
  dropdown — it must read as a peer setting, not a bespoke widget.
- **Option labels:** endonyms in each language's own script (Español, 日本語, 한국어,
  简体中文, Русский …) so a user recognizes their language without reading English. CJK
  glyphs fall back to a system face within the dropdown — acceptable for option text.
- **Colour & tokens:** inherits the `Select` primitive's tokens — `--surface-*`/glass
  background, `--text` / `--text-muted`, `--accent` focus ring, control height and radius
  tokens from `@lunma/tokens`. No hardcoded values; honours the active `data-tint`
  substrate like every other control.
- **Interaction feedback:** native `<select>` focus/hover/active/open states via the
  primitive's existing styling; visible `--accent` focus ring; the selected value is the
  current locale on load.
- **Motion:** selection persists immediately (no save button), then the surface reloads
  to apply the new locale — there is no in-place tween for the switch itself in this
  change (see D6). The dropdown's own open/close uses the `Select` primitive's existing
  timing; reduced-motion and WCAG-AA contrast hold because no new motion or colour is
  introduced.
- **Arc reference / divergence:** Arc surfaces language under OS-level settings; Lunma
  keeps it in-app next to the other Appearance controls so the choice is discoverable and
  immediate, consistent with Lunma's "settings are one declarative surface" approach.

## Risks / Trade-offs

- **Generated code in `shared/` could violate the DAG** → it imports nothing from other
  layers by construction (legal by construction). Note Biome does **not** enforce this on
  the generated dir, since `tasks.md` 1.3 excludes it from linting; the guarantee rests on
  Paraglide's output shape, and `biome check`'s `noImportCycles`/`noRestrictedImports`
  still run on the *consuming* code (`i18n.ts`), which is the edge that could actually
  break the DAG.
- **Custom strategy must be registered before any message evaluates** → Paraglide v2
  throws if `getLocale()` runs before `defineCustomClientStrategy('custom-lunmaSettings')`.
  `i18n.ts` registers at module-load (import-for-side-effect); latent in this change (no
  surface strings are consumed yet) but specified now for the consumer change.
- **SW crash from a window-touching strategy** → mitigated by restricting the strategy
  array to `custom + baseLocale` (D3); a regression would surface immediately as a SW
  load error. Consider a guard test that the resolver references no `window`/`document`.
- **`settings ↔ i18n` import cycle** → avoided by literal endonym labels (D4); `biome
  check`'s `noImportCycles` is the backstop.
- **Reload feels heavy for a language switch** → acceptable for a rare, deliberate action
  and stateless on boot; in-place re-render is a noted future polish (D6).
- **Committed generated output adds diff noise** → bounded; matches the accepted
  `icon-loaders.generated.ts` trade-off; the parity test guards catalog drift.
- **Manifest `_locales` drift from Paraglide messages** → they are independent catalogs;
  the parity test covers both `messages/` and `_locales/` key completeness.
- **`auto` → locale mapping gaps** (e.g. `pt-BR` browser → `pt-PT`) → explicit base-tag
  mapping with `en` as the terminal fallback; covered by spec scenarios.

## Migration Plan

No data migration: `language` is a new optional setting defaulting to `'auto'`; absent
stored values resolve to defaults via the existing `.catch()` engine. Rollback = revert
the change; no persisted state needs cleanup (a stored `language` value is simply ignored
by older code, which never reads the key). Ship order: this foundation →
`localize-extension-ui` (string migration + real translations) → `localize-marketing-site`.
