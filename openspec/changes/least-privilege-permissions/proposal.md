## Why

Lunma's first public Chrome Web Store release currently asks every new user to grant — at install, before they've done anything — access to **all websites**, their **entire browsing history**, and **all bookmarks**. That is an alarming, trust-eroding first impression for a workspace tool, and it is broader than most users' actual usage warrants: history and bookmarks are only read to answer launcher queries, and broad website access is only needed by the smart-folder connectors when a user actually connects a service. This change moves those to least-privilege: the install prompt asks only for what the core workspace needs, and each optional capability is granted in-context, the first time the user reaches for the feature, via an inline "Enable" control.

The directly visible value is a calmer, more trustworthy install and a feature-by-feature consent model the user controls. The named downstream consumer is the **Chrome Web Store first public release** (`chrome-store-submission`, the launch this unblocks) — least-privilege materially reduces reviewer friction around the scariest line item (blanket fetch access to any site, including arbitrary self-hosted hosts) and produces a cleaner store-listing privacy narrative.

**Honest scope limit (carried into design):** the launcher overlay and pinned-tab boundary content scripts must remain on `<all_urls>` at `document_start` (the launcher must be summonable on any page; the boundary must catch clicks before navigation). Because broad content-script `matches` alone keep the item in Chrome's "broad host permissions" manual-review tier, this change does **not** escape manual review. Its value is least-privilege posture, a cleaner install prompt, and connector hosts requested only when used — not avoiding review.

## What Changes

- **Manifest (`apps/extension/public/manifest.json`):**
  - **BREAKING (permission grant model):** Remove `<all_urls>` from `host_permissions`. Connector host access is requested at runtime.
  - Add `optional_host_permissions` covering connector hosts (specific `https://github.com/*`, `https://api.github.com/*`, `https://gitlab.com/*`, `https://*.atlassian.net/*`, plus broad `https://*/*` + `http://*/*` patterns for arbitrary self-hosted GitLab/GitHub Enterprise/Jira `baseUrl`s).
  - Move `history` and `bookmarks` from `permissions` to `optional_permissions`.
  - Drop `favicon` **iff** a verification step confirms Chrome 123+ still populates `tab.favIconUrl` and the `_favicon/*` web-accessible path without it; otherwise keep it and record why in design.
  - The two `content_scripts` (`launcher/overlay.ts`, `content/tab-boundary.ts`) stay on `<all_urls>`; `tabs`, `tabGroups`, `storage`, `sidePanel`, `alarms`, `scripting`, `commands` stay as required permissions.
- **New `shared/permissions.ts` module:** a thin, typed wrapper over `chrome.permissions` — query/request/remove for both API permissions (`history`, `bookmarks`) and host origins (derived from a connector `baseUrl`), plus a subscribe-to-changes surface over `chrome.permissions.onAdded`/`onRemoved` so UI reacts to grants/revocations without a reload.
- **Connectors layer:** each connector declares the origin(s) it actually fetches — GitHub on `github.com` fetches `api.github.com` (a different origin), GitLab/Jira/GHE fetch same-origin under `baseUrl`, and RSS fetches the feed URL itself. A folder fetches only when **all** of its connector's required origins are granted; otherwise it resolves to a distinct `needs-access` runtime state (not a generic `error`). Creating/enabling a folder requests that origin set. This applies to **all four** sources, including RSS (a feed on an ungranted origin shows `needs-access` rather than failing).
- **Launcher:** the history and bookmarks suggestion providers degrade gracefully — when their permission isn't granted, the launcher returns its other sources (tabs, search, navigation) and surfaces an inline "Enable history/bookmark results" affordance instead of failing.
- **UI (inline grant affordance):** smart-folder surfaces and the launcher render an "Enable / Grant access to `<host>`" control that calls `chrome.permissions.request()` on click and is reversible; it composes existing primitives (no re-rolled buttons).
- **Docs + specs:** updated in lockstep (see Impact).

## Capabilities

### New Capabilities
- `runtime-permissions`: The optional-permission grant model — which permissions are required vs optional in the manifest, the `shared/permissions.ts` query/request/remove/subscribe contract, and the inline grant-affordance pattern (a feature surface composing the `Button`/`Icon` primitives, never a generic install-time wall).

### Modified Capabilities
- `smart-folders`: connector fetches are gated on a runtime grant of the connector's required origins; an ungranted origin resolves to a `needs-access` runtime state; folder creation/enable requests that origin set; the smart-folder card and editor render the grant affordance (the options Connectors section is unchanged — token management only).
- `launcher`: the shared `history` and `bookmarks` data providers become permission-gated and degrade gracefully; the launcher surfaces an inline affordance to grant them rather than treating absence as an error.

## Impact

- **Code:** `apps/extension/public/manifest.json`; new `apps/extension/src/shared/permissions.ts`; `apps/extension/src/background/connectors/{connector,github,gitlab,jira,rss}.ts` (each declares its required fetch origins) and `smart-folders.ts` (gate fetch on the connector's required origins + `needs-access` state, handling both grant and revoke directions); `apps/extension/src/background/launcher-suggestions-handler.ts` (gate the `chrome.history`/`chrome.bookmarks` calls — the launcher's pure providers under `launcher/shared/` are unchanged); UI surfaces `apps/extension/src/sidebar/SmartFolder.svelte`, `SmartFolderEditor.svelte`, and the launcher result surface (grant affordance). The options **Connectors** section is **unchanged** (token management only), but the options page gains a **new, separate "Result sources" section** (`#result-sources`) — the inline history/bookmarks grant control and the location the Alt+L overlay deep-links to via the new `background/open-options-grant.ts` SW helper (agreed via AskUserQuestion: the overlay can't call `chrome.permissions`, so its "Enable" must route somewhere it can be granted).
- **Public API introduced (normative — additions beyond these require agreement + artifact update):** `shared/permissions.ts` exports — `hasApiPermission(name)`, `requestApiPermission(name)`, `hasHostPermissions(origins: string[])` (true only if every origin is granted), `requestHostPermissions(origins: string[])`, `originPatternForBaseUrl(baseUrl)`, and `onPermissionsChange(listener)`; a new `requiredOrigins(node): string[]` member on the `SourceConnector` contract (`background/connectors/connector.ts`); a pure `requiredOriginsForNode(node)` helper in `shared/connector-origins.ts` — the single origin derivation the `SourceConnector.requiredOrigins` members delegate to AND the sidebar/editor surfaces call directly (a surface cannot import `background/connectors` under the layer DAG, so the derivation lives in `shared/`, mirroring the `launcher/shared` pure-provider precedent); a `'needs-access'` member added to the `SmartFolderRuntime['state']` union in `shared/types.ts`; an optional `ungrantedSources?: ('history' | 'bookmarks')[]` field on `SuggestionsResult` / the suggestions response message (SW-computed, so the stateless overlay knows which sources to offer — agreed via AskUserQuestion); and a `lunma/open-options-grant` message (overlay → SW) routing the overlay's "Enable" affordance to the options grant control.
- **UI primitives:** composes existing `Button`, `IconButton`, `Icon`, `Surface`; **no new primitives** planned. If design finds one is needed, that is a deviation to surface before it lands.
- **`docs/`:** updates `docs/03-architecture.md` (permission/grant model + the load-bearing-content-script constraint) and `docs/04-capabilities.md` (new `runtime-permissions` capability; smart-folders + launcher degradation notes). Leaves `docs/01-vision.md`, `02-tech-stack.md`, `05-roadmap.md`, `06-migration.md`, `distribution.md`, `07-brand-identity.md` untouched.
- **Out of scope (separate change):** the marketing-site `/privacy` route (a `marketing-site` capability change) and the Chrome Web Store dashboard listing/justification text (operational, not code).
- **Dependencies:** none added; uses the `chrome.permissions` MV3 API already available under the toolchain.
- **Testing:** unit tests for `shared/permissions.ts` and the gated connector/provider paths; existing smart-folder and launcher tests updated for the new state/degradation; e2e smoke unaffected (granting in Playwright is handled via the test harness).
