## Why

A smart folder is already a work/reading hub — N sources × M filters plus RSS feeds, all aggregated in the broadcast-only `smartFolders` runtime slice — but it can only be seen through the ~300px sidebar, where "one-glyph restraint" squeezes every section into a thin column of link rows. This change gives a smart folder a **dedicated full-page view**: a spacious, immersive command-center that renders *all* of the folder's sections at once — room to breathe, the active Space's aurora, the work and the reading laid out editorially. The user-visible value is immediate: a folder you only glanced at in the sidebar becomes a place you can actually sit in and scan your whole day's work or reading at a glance, opened in its own tab.

The page is the proven `NewTab.svelte` recipe — a read-only mirror of SW state, composing existing `ui/` primitives and the `@lunma/tokens` aurora/glass/hearth recipes — pointed at a single folder, with no new persisted results and no new host permissions. The page's item unit is a **card with optional content slots**, and this change also **fills those slots for RSS feeds**: each feed entry becomes a magazine-style card with its **description excerpt, thumbnail image, and publication date** (the RSS connector already fetches the feed; it now parses these fields too). Queue sources (GitLab/GitHub/Jira) have no description/image, so per-item *queue* enrichment (MR diff stats, CI detail) stays out of scope — a future `smart-folder-page-rich-content` change can fill those same slots for queues, purely additively, with no surface rewrite (the **B-seam** below).

## What Changes

- **New full-page surface** — a standalone extension page (its own `index.html` + `main.ts` + Svelte component, registered in the build/manifest the same way `launcher/newtab` is) reachable at a `chrome-extension://` URL carrying the target `folderId` (and `windowId`) as URL params. It is a read-only mirror of SW state via `onStateBroadcast`, exactly like `NewTab.svelte`.
- **Spacious sectioned dashboard** — the page renders every **resolved section** of the folder (one per source × filter, plus each rss feed) in an editorial multi-column layout. It reuses the existing per-kind row semantics (queue = status dots; feed = unread marks) and the calm per-section `pending` / `error` / `signed-out` / `needs-access` / empty states — the same projection rules as the sidebar, given room.
- **Item unit is a card with optional slots (the B-seam)** — each result renders as a card whose layout reserves regions for richer content: a hero image, an excerpt, and a meta/date footer. Queue items fill only title + favicon + status (clean, uncrowded); **feed items fill the slots** (see next bullet). The optional `SmartFolderItem` fields (`excerpt?`, `imageUrl?`, `publishedAt?`) are additive and ride the **ephemeral** runtime slice, so there is **no schema migration**; the sidebar projection ignores them.
- **Rich RSS cards** — the RSS connector now parses each entry's **description** (HTML stripped to a clamped plain-text excerpt), **image** (`media:content` / `media:thumbnail` / `enclosure`, else the first inline `<img>`), and **publication date** (`pubDate` / Atom `published`/`updated`). The page renders feed sections as a full-width **magazine grid** of cards: hero image on top, title, excerpt, and a relative date. Thumbnails load **lazily** and with **`referrerpolicy="no-referrer"`** so the publisher gets no referrer (the residual cost — the publisher's server sees the request IP when an image loads — is documented in design).
- **Three sidebar entry points:**
  1. An **"Open as page"** action in the smart folder's pinned-header (kebab) menu.
  2. **Activating the folder's label/body** opens the page, while the **disclosure chevron keeps expand/collapse** — a deliberate gesture split (diverging from a regular folder row, whose body-click toggles expand) that resolves the open-vs-collapse confusion; documented in design.
  3. The page opens as a **reused tab per window**: reopening focuses the existing page tab rather than spawning duplicates (achieved by tab-query dedupe, not a persisted binding — see design D2).

  A fourth reach — a **launcher result** that opens the folder's page — is deferred to a named follow-up change `smart-folder-page-launcher`, because it reverses the launcher's existing "folders are never emitted as result rows" requirement and deserves its own proposal/review. This change is the upstream that follow-up consumes (it ships `openSmartFolderPage`, which the launcher result will dispatch).
- **Activation reuses existing bus commands** — result rows open via the existing `openSmartItem` / `openUrl` semantics; feed read-state semantics are unchanged. The page **never mutates state directly**.

Non-goals (explicitly out of scope for this change): per-item **queue** enrichment (MR/PR diff stats, CI detail — feed enrichment IS in scope), any new connector network fetch (the RSS connector parses richer fields from the response it already fetches — no extra request), an "all folders / all my work" aggregate page, and persisting results.

## Impact

- **New files (extension):** a new page directory under `apps/extension/src/` (following the `launcher/newtab` precedent) — `index.html`, `main.ts`, the page Svelte component, its CSS, and tests. Exact path settled in design.
- **Build/manifest:** register the new HTML entrypoint via `build.rollupOptions.input` so crxjs builds it and it's reachable at a `chrome-extension://` URL. **Not** added to `web_accessible_resources` (the extension opens its own page; WAR would over-expose a state-mirroring page to all web origins — see design D1).
- **Background:** a new `openSmartFolderPage` command that opens-or-focuses the page tab per window by **querying open tabs for the page URL + folderId** (no persisted binding, self-healing across SW restarts); no connector or polling change. Note: the page opens via this command, **not** `openUrl` (whose scheme hardening drops the `chrome-extension://` page URL by design and is left untouched).
- **Sidebar:** the smart folder header gains the gesture split (`FolderRow` gains an optional `onActivate`) + "Open as page" menu item + a hover-revealed "open as page" icon button; dispatches the new open command.
- **Launcher:** untouched by this change (the page-open launcher result is the deferred `smart-folder-page-launcher` follow-up).
- **RSS connector + types:** `apps/extension/src/background/connectors/rss.ts` parses description/image/date from the already-fetched feed body; `SmartFolderItem` (`shared/types.ts`) gains optional `excerpt?`/`imageUrl?`/`publishedAt?`; the ephemeral `SmartFolderItemSchema` mirror (`shared/schemas.ts`) gains the same optional fields. No persisted-shape change (the slice is stripped before persist).
- **`ui/` primitives — composed (no re-roll):** `Surface` (glass panels), `Aurora` (backdrop), `Icon`, `FaviconTile`/favicon helpers, `Button`, `Chip`, `Kbd` — the same set `NewTab.svelte` composes. **New primitives:** none anticipated; if a genuinely cross-surface card primitive proves necessary it ships in this change with its contract proven by use (settled in design). The page's own section/card layout is a feature component, not a primitive.
- **Docs:** updates `docs/architecture.md` (the new surface in the layer map / surface list). Leaves `docs/tech-stack.md` untouched (no new dependency). The `smart-folders` and `launcher` capability specs are updated via this change's delta files on archive.
- **No new dependencies, no new host permissions, no new persisted result state.**
