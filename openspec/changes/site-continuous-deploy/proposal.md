## Why

This change puts **`lunma.app` on the internet**: the already-built marketing
site (`apps/site` — hero, trust band, FAQ, the `/privacy` policy, the OG card,
the sitemap) goes from "passes `vite build` on the author's machine" to a live,
publicly reachable page served from this repo's CI on every push to `main`. The
user-visible value is the landing page itself — the first thing a prospective
user sees, and the public face the whole "local-first, honest, open" positioning
rests on.

It is also load-bearing plumbing for a *named* downstream change: a Chrome Web
Store listing **requires** a reachable privacy-policy URL, and Lunma's is
`https://lunma.app/privacy`. Until the site is deployed, that field cannot be
filled, so `extension-release-pipeline` (the next change in the sequence) cannot
submit the listing. This change is the smallest plumbing that unblocks it — and
unlike pure infrastructure it *also* ships the visible site, so it satisfies both
shapes of the user-value policy at once.

This is the **second** of the four sequenced release-engineering changes named in
`github-repo-and-ci` (`site-continuous-deploy`, `extension-release-pipeline`,
then `open-source-public-launch`). It runs while the repo is still **private**
(the public flip is the last change), so the deploy must work from a private
repo — which, like `github-repo-and-ci`'s deferred branch protection, rules out
GitHub Pages (paid for private repos on a Free org) and is exactly why Cloudflare
Pages is the host.

## What Changes

- **A CI-driven Cloudflare Pages deploy.** A new `.github/workflows/deploy.yml`
  runs on push to `main`: it builds the site with the repo-pinned toolchain (the
  same Node 24 / corepack-`pnpm@11.3.0` / pnpm-store-cache setup `ci.yml` already
  uses), then publishes the static `apps/site/build/` output to Cloudflare Pages
  via **`wrangler pages deploy`**. The build env is Lunma's own pinned one — *not*
  Cloudflare's Git-integration build runners (decision D2) — so "deploys
  `lunma.app` from this repo's CI" is literally true and there is no second,
  drifting build environment.
- **A separate workflow, not a job on `ci.yml`.** Deploy needs a write-scoped
  Cloudflare token; `ci.yml`'s `verify`/`e2e` jobs stay strictly
  `permissions: contents: read`. Keeping deploy in its own file preserves that
  least-privilege split (D1).
- **Production on `main`, previews on same-repo branches.** A push to `main`
  publishes to the production `lunma.app` deployment. Pushes to other branches in
  *this* repo publish a Cloudflare preview deployment (`*.lunma.pages.dev`). The
  deploy workflow does **not** run on `pull_request` events — so a fork PR (once
  the repo is public in change #4) can never receive the Cloudflare secret (D3).
- **Custom-domain binding.** The Cloudflare Pages project (`lunma`) is bound to
  the `lunma.app` apex (and `www` → apex redirect) using the already-on-Cloudflare
  DNS for the registered domain. Captured as exact dashboard/`wrangler` steps in
  `tasks.md` (account state, not a repo file).
- **Two GitHub repo secrets:** `CLOUDFLARE_API_TOKEN` (scoped to **Pages:Edit
  only** — least privilege) and `CLOUDFLARE_ACCOUNT_ID`. Recorded as `gh secret
  set` commands in `tasks.md`; never committed.
- **Post-deploy smoke.** After publishing, the workflow asserts the deploy is
  actually live and the load-bearing URL resolves — `curl --fail` against
  `https://lunma.app/` and `https://lunma.app/privacy` expecting `200` — so a
  broken publish fails the run instead of silently serving stale content.
- **Hardened response headers (CSP included).** The served site sends a
  defense-in-depth header set so it is secure from first paint, matching Lunma's
  privacy-forward positioning. A new `apps/site/static/_headers` (copied to the
  Pages build root) carries `Strict-Transport-Security`, `X-Content-Type-Options:
  nosniff`, `Referrer-Policy`, `X-Frame-Options: DENY` (clickjacking), and
  `Permissions-Policy`. A **strict, same-origin Content-Security-Policy** is
  generated by SvelteKit's `kit.csp` in **`hash` mode** (configured in
  `apps/site/svelte.config.js`) — the only prerender-compatible CSP mode, which
  auto-hashes SvelteKit's own inline hydration script/style each build and emits
  the policy via `<meta>`, so no `'unsafe-inline'` for scripts and no
  hand-maintained hashes (D5). The exact directive set (notably whether
  `style-src` needs a documented relaxation for any Svelte-emitted inline `style`
  attributes) is finalized at apply time against the built output and proven by a
  post-deploy no-violations check.
- **No re-gating of `verify`.** Merges to `main` are fast-forwards of
  already-green commits (`github-repo-and-ci` §5.3 workflow), and `ci.yml`'s
  `push: [main]` re-runs `verify` independently; the deploy trusts that gate
  rather than hard-coupling to it via `workflow_run` (D4).

The site continues to ship in its honest pre-launch state — `LAUNCHED = false` in
`apps/site/src/lib/links.ts` keeps the install CTAs in their "coming soon" form
(no store listing exists yet); flipping that flag is `extension-release-pipeline`
/ launch work, **not** this change.

**Out of scope (named, deferred):**
- Fine-grained cache-control tuning beyond Cloudflare Pages' defaults — Vite's
  content-hashed asset filenames already get immutable caching; bespoke
  cache-control headers are not worth the risk here. (The security headers above
  *are* in scope; only cache tuning is deferred.)
- Extension build/version/zip, GitHub Releases, the Chrome Web Store submission
  itself → `extension-release-pipeline`.
- The private→public repo flip, DCO, and finally-enforced branch protection →
  `open-source-public-launch`.

This change ships **no `src/` runtime code** and authors **no new visual
language**: the site's surface and its visual bar were delivered in the
`marketing-site` changes; this change only *publishes* that already-to-bar
surface (its only `apps/site` touches are build/deploy config — the `kit.csp`
block in `svelte.config.js` and the `static/_headers` file — not `src/` or
`src/ui`). So there is no `Visual language` section and no `src/ui/` primitive
work — stated here as an explicit exemption (per the visual-quality and
component-library policies), the same way the sibling release-engineering changes
do. No `src/ui/` primitives are composed or added (none apply).

### Modified Capabilities
- `release-engineering`: **ADDS** the continuous-deployment requirements to the
  capability `github-repo-and-ci` introduces — that the marketing site is built
  with the repo-pinned toolchain and published to its production host on every
  push to `main`, that production deploys only from `main` while previews come
  from same-repo branches (never fork PRs), that deploy credentials are
  least-privilege and never exposed to forks, that a deploy is verified live
  (the `/privacy` URL resolves) before it is considered done, and that the served
  site sends a hardened response-header set (HSTS, nosniff, frame protection,
  Referrer/Permissions policy, and a strict same-origin CSP). (The base
  `release-engineering` spec becomes living when `github-repo-and-ci` archives;
  this delta is additive and lands when this change archives — a sequencing note,
  not a conflict. See design.)

## Impact

- **New files:**
  - `.github/workflows/deploy.yml` — the CI-driven Cloudflare Pages deploy.
  - `apps/site/static/_headers` — Cloudflare Pages response headers (HSTS,
    nosniff, `Referrer-Policy`, `X-Frame-Options: DENY`, `Permissions-Policy`);
    copied to the build root by adapter-static.
- **Modified files (build/deploy config only, no `src/` runtime code):**
  - `apps/site/svelte.config.js` — add the `kit.csp` (hash-mode) block that
    generates the strict same-origin Content-Security-Policy.
- **New OpenSpec spec delta:** `openspec/changes/site-continuous-deploy/specs/release-engineering/spec.md`
  (`## ADDED Requirements`), folding into `openspec/specs/release-engineering/`
  on archive.
- **External / account state (not files), recorded as reproducible commands in
  `tasks.md`:**
  - the Cloudflare Pages project `lunma`;
  - the `lunma.app` apex (+ `www` redirect) custom-domain binding on the
    already-Cloudflare-managed DNS;
  - the two GitHub repo secrets `CLOUDFLARE_API_TOKEN` (Pages:Edit) and
    `CLOUDFLARE_ACCOUNT_ID`.
- **Docs updated in this change (doc-lockstep):**
  - `docs/tech-stack.md` — extend the CI/release-engineering note: the site is
    deployed to Cloudflare Pages from CI via `wrangler` on the same pinned
    toolchain; production from `main`, previews from same-repo branches.
  - `docs/architecture.md` — a short "Continuous deployment" subsection alongside
    the existing "Continuous integration" one: `apps/site` → `wrangler pages
    deploy` → `lunma.app`, build-time-only, nothing in the extension bundle; note
    the served-site hardening (CSP via `kit.csp` hash mode + `static/_headers`).
- **Docs explicitly left untouched:** the capability specs under
  `openspec/specs/` (changed only on archive); the ADRs; `README.md` (no
  contributor- or content-facing change here — the public-launch README edits
  belong to `open-source-public-launch`).
- **Dependencies:** **none added to any `package.json`** — preserving
  `github-repo-and-ci`'s zero-new-deps posture. `wrangler` runs via the official
  **`cloudflare/wrangler-action`**, pinned to a major tag/SHA exactly like the
  workflow's other third-party actions (D6); its version lives in the workflow
  pin, dependabot-tracked through the `github-actions` ecosystem already declared
  in `.github/dependabot.yml`.
- **No new `host_permissions`, manifest, or runtime surface.**
