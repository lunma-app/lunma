## Requirements
### Requirement: Supported locales and the message-catalog contract

Lunma SHALL define a fixed set of supported UI locales: `en` (the base locale and
source of truth) plus `es`, `pt-PT`, `fr`, `de`, `ja`, `ko`, `zh-CN`, `ru`. The set
SHALL be declared once in `apps/extension/project.inlang/settings.json`
(`baseLocale: "en"`, `locales: [...]`) and mirrored by the `SupportedLocale` union type
exported from `apps/extension/src/shared/settings.ts`.

UI message strings SHALL live in per-locale catalog files
`apps/extension/messages/{locale}.json` (one file per supported locale), outside
`apps/extension/src/` so they are excluded from the TypeScript, Biome, and Stylelint
globs. `en.json` SHALL be the authoritative source; until a downstream change supplies
translations, the other locale files MAY contain English copies, but every file SHALL
carry the same key set (see the parity requirement).

Messages SHALL be consumed only through the compiled Paraglide runtime generated into
`apps/extension/src/shared/paraglide/`, which is regenerated by a `gen:i18n` script
(`paraglide-js compile`) and committed to the repository (mirroring the existing
`src/ui/icon-loaders.generated.ts` generated-and-committed precedent). The generated
directory SHALL be excluded from Biome linting. Because it lives under `shared/`, the
generated runtime SHALL import nothing from other `apps/extension/src/` layers, keeping
the import DAG legal.

#### Scenario: The locale set is single-sourced

- **WHEN** the supported-locale set is needed by the inlang project and by the `SupportedLocale` type
- **THEN** both SHALL list exactly `en, es, pt-PT, fr, de, ja, ko, zh-CN, ru` with `en` as the base

#### Scenario: Generated runtime keeps the layer DAG legal

- **WHEN** `biome check` runs over `apps/extension/src/shared/paraglide/`-consuming code
- **THEN** no import from `ui`, `background`, `sidebar`, `launcher`, `options`, or `content` into `shared/` SHALL be introduced by the i18n runtime, and `biome check` SHALL pass

### Requirement: Service-worker-safe, settings-backed locale resolution

Locale resolution SHALL be exposed through a single module
`apps/extension/src/shared/i18n.ts` — the source of truth for **locale state**
(resolution, persistence, the active locale). Surfaces SHALL obtain locale state only via
`i18n.ts`, never by calling the generated Paraglide runtime's locale APIs
(`getLocale`/`setLocale`/strategy internals) directly. Surfaces MAY import message
functions (`m.*`) from the generated runtime to render strings — that is the intended
Paraglide usage in the downstream `localize-extension-ui` change; the constraint is on
locale *control*, not on message *rendering*. `i18n.ts` SHALL register its custom
strategy at module load (an import-for-side-effect), so the strategy is registered before
any message function evaluates `getLocale()`.

The module SHALL register a custom Paraglide client strategy named
`custom-lunmaSettings` whose synchronous `getLocale()` returns an in-memory cached
locale and whose `setLocale(locale)` persists the choice via
`writeSetting('language', locale)`. The Paraglide strategy array SHALL be
`['custom-lunmaSettings', 'baseLocale']` **only** — the default `url`, `cookie`,
`localStorage`, and `preferredLanguage` strategies reference `window`/`document`/
`localStorage` and SHALL NOT be used, because they would throw when the `shared/` bundle
is loaded in the service-worker context.

The module SHALL export `initLocale()`, `applyLocaleFromSettings(language)`,
`getLocale` (re-exported from the generated runtime), and `setLocale`. `setLocale`
SHALL be a thin typed wrapper — not a raw re-export — because the user-facing
`'auto'` sentinel is not a Paraglide `Locale` and the runtime export's type rejects
it; the wrapper accepts `SupportedLocale | 'auto'`, persists the choice via
`writeSetting('language', value)`, re-seeds the cache, and (since the gated
`watchSettings` path owns the reload, design D6) never reloads. `initLocale()` SHALL read the persisted `language`
setting; when it is `'auto'` (or unset) it SHALL resolve the locale from the browser via
`chrome.i18n.getUILanguage()` (falling back to `navigator.language`), mapping a
base-language tag to the nearest supported locale (e.g. `pt → pt-PT`, `zh → zh-CN`),
and SHALL fall back to the base locale `en` when no supported match exists. When the
cache is unseeded (e.g. under jsdom in unit tests, where `chrome.storage` is absent),
`getLocale()` returning `undefined` SHALL fall through to `baseLocale` (`en`).

#### Scenario: Resolver is the only localization entry point

- **WHEN** a surface needs the active locale or a message
- **THEN** it SHALL import from `src/shared/i18n.ts` (or message functions seeded by it), not from the generated runtime directly

#### Scenario: First run with no stored language resolves from the browser

- **GIVEN** no `language` has been stored (effective value `'auto'`) and the browser UI language is `de`
- **WHEN** `initLocale()` runs
- **THEN** the resolved locale SHALL be `de`

#### Scenario: A base-language tag maps to the nearest supported locale

- **GIVEN** effective `language` is `'auto'` and the browser UI language is `pt-BR`
- **WHEN** `initLocale()` runs
- **THEN** the resolved locale SHALL be `pt-PT` (nearest supported), not the base locale
- **AND** a `zh-TW` (Traditional) browser SHALL likewise map to `zh-CN` — the only
  supported Chinese variant (a documented, accepted lossiness, not a silent gap)

#### Scenario: An unsupported browser locale falls back to base

- **GIVEN** effective `language` is `'auto'` and the browser UI language is `th` (unsupported)
- **WHEN** `initLocale()` runs
- **THEN** the resolved locale SHALL be `en`

#### Scenario: Service-worker bundle does not touch window/document for locale

- **WHEN** the `shared/` bundle (including the i18n runtime) is evaluated in the service worker
- **THEN** locale resolution SHALL NOT reference `window`, `document`, or `localStorage`, and SHALL NOT throw

#### Scenario: Tests with no chrome.storage render in the base locale

- **WHEN** a unit test renders a localized surface under jsdom without seeding the cache
- **THEN** `getLocale()` SHALL fall through to `baseLocale` and messages SHALL resolve in English

### Requirement: Surfaces seed the locale before first paint and re-apply on change

Each document surface SHALL seed the locale cache by calling `initLocale()` before its
first paint. `sidebar/main.ts` and `launcher/newtab/main.ts` already run an async
`boot()` that `await readSettings()` before `mount()`; `initLocale()` SHALL be added to
that existing seam. `options/main.ts` currently mounts synchronously (it has no
pre-`mount()` settings read); it SHALL be restructured into an async boot that
`await`s `initLocale()` (and `readSettings()`) before `mount()`, so the Options first
paint is already localized. (This restructure of `options/main.ts` is the one piece of
new, surface-local boot machinery this change introduces.)

When the `language` setting changes, each open surface's `watchSettings` callback SHALL
call `applyLocaleFromSettings(newLanguage)` and reload the surface (`location.reload()`).
The reload SHALL be gated on a **change to `language` specifically** — the callback SHALL
compare the previous and new `language` and reload only when it differs; changes to any
other setting (density, theme, tint, glares, …) SHALL continue to apply **live, without a
reload**, exactly as before this change. `Options.svelte`, which has no `watchSettings`
subscription today, SHALL gain one for this gated reload path; `sidebar/main.ts` and
`launcher/newtab/main.ts` SHALL extend their existing `watchSettings` callbacks with the
gated language branch. The surface that initiates the change SHALL call `setLocale` with
`{ reload: false }` so the gated `watchSettings` path is the single owner of the reload
(no double reload). The launcher overlay, dormant between opens, SHALL pick up the current
locale when it next opens rather than via a persistent subscription.

#### Scenario: Initial paint is already localized

- **GIVEN** the stored `language` resolves to `fr`
- **WHEN** a surface boots
- **THEN** `initLocale()` SHALL have seeded the cache before `mount()`, so the first rendered frame is in French

#### Scenario: Options is localized on first paint via its async boot

- **GIVEN** the stored `language` resolves to `fr`
- **WHEN** the options page boots
- **THEN** `options/main.ts` SHALL `await initLocale()` before `mount()`, so the Options first paint is already French (not an English-then-French flash)

#### Scenario: Changing language re-localizes open surfaces

- **WHEN** the user selects a different language in options
- **THEN** the choice SHALL persist via `setLocale` (`{ reload: false }`) / `writeSetting`
- **AND** each open surface's `watchSettings` SHALL detect the `language` delta, apply the new locale, and reload, with no double reload on the initiating surface

#### Scenario: A non-language settings change does not reload

- **WHEN** the user changes a non-language setting (e.g. density or colour intensity)
- **THEN** each surface's `watchSettings` SHALL apply it live with NO `location.reload()`, exactly as before this change

### Requirement: Native manifest and store-listing localization

The extension manifest SHALL be localized via the native `chrome.i18n` mechanism,
independent of Paraglide. `apps/extension/public/manifest.json` SHALL declare
`"default_locale": "en"` and SHALL reference localizable values as `__MSG_key__`
placeholders for: the extension `description`, `action.default_title`, and the
`description` of each declared command (`pin-active-tab`, `toggle-launcher`). The brand
`name` and `short_name` ("Lunma") SHALL remain literal and untranslated.

For every supported locale there SHALL be a
`apps/extension/public/_locales/{locale}/messages.json` providing the referenced message
keys; these files SHALL be passed through verbatim by the build (as `public/` already is
for `fonts/` and `icons/`). The `_locales` subdirectory names SHALL use Chrome's
underscore locale codes (`pt_PT`, `zh_CN`), not the BCP-47 hyphens the Paraglide
`messages/` catalogs use — a platform path convention, mapped (`-` → `_`) by the parity
and locale-set tests. The Chrome Web Store listing's title/description fields are
localized in the CWS dashboard (an external, non-code task) and are out of scope for this
spec.

#### Scenario: Manifest declares a default locale and message placeholders

- **WHEN** the built manifest is inspected
- **THEN** it SHALL contain `default_locale: "en"` and `__MSG_*__` placeholders for the description, action title, and command descriptions, while `name`/`short_name` remain the literal string "Lunma"

#### Scenario: A non-English UI localizes the action title

- **GIVEN** the browser UI language is a supported non-English locale with a populated `_locales` entry
- **WHEN** the extension loads
- **THEN** `chrome.i18n` SHALL resolve the action default title and description from that locale's `messages.json`

### Requirement: Locale catalogs are key-complete (parity test)

An automated test `apps/extension/src/i18n-parity.test.ts` SHALL assert that every
Paraglide message catalog (`apps/extension/messages/{locale}.json`) and every manifest
locale catalog (`apps/extension/public/_locales/{locale}/messages.json`) carries exactly
the same key set as its `en` counterpart, with no empty string values. The catalog files
SHALL be read at test runtime (e.g. `readFileSync`), not via static `import`, so they
remain outside the TypeScript/Biome/Stylelint graph.

#### Scenario: A missing key fails the parity test

- **WHEN** a non-`en` catalog is missing a key present in `en` (or contains an empty value)
- **THEN** `i18n-parity.test.ts` SHALL fail

#### Scenario: Complete catalogs pass

- **WHEN** every locale catalog has the same keys as `en` with non-empty values
- **THEN** the parity test SHALL pass

### Requirement: User-facing surface strings render through the message catalog

Every user-visible string in the sidebar, launcher (new-tab + the `Alt+L` overlay), and options surfaces SHALL render through a Paraglide message function (`m.*`), not a hardcoded literal. New message keys SHALL be added to
`apps/extension/messages/en.json` (the source of truth) under a flat, surface-namespaced
key convention; interpolation and plurals SHALL use Paraglide's ICU message syntax.
Surfaces SHALL continue to obtain locale *state* only via `src/shared/i18n.ts`; they
render messages via the generated `m.*` directly (the locale-state-vs-message-rendering
split established by `add-i18n-foundation`).

The brand string "Lunma" and the endonym option labels in `settings.ts` SHALL remain
literal (intentionally untranslated).

#### Scenario: A surface string is read from the catalog

- **WHEN** a sidebar/launcher/options surface renders a user-facing label
- **THEN** the text SHALL come from a `m.*` message function keyed in `messages/en.json`, not a string literal in the component

#### Scenario: Switching locale changes the rendered text

- **GIVEN** the `language` setting resolves to `de`
- **WHEN** a migrated surface paints
- **THEN** its strings SHALL render from the `de` catalog (German), not English

### Requirement: Authored translations for every supported non-base locale

Each supported non-base locale SHALL ship authored, context- and length-conscious translations of every key — replacing the English-copy seeds from `add-i18n-foundation` — in both `apps/extension/messages/{locale}.json` and `apps/extension/public/_locales/{locale}/messages.json`. The existing catalog-parity test SHALL continue
to guarantee key-completeness with no empty values; translation *quality* is a human
review gate (not asserted by a test).

#### Scenario: No locale ships English placeholders for migrated keys

- **WHEN** a migrated key is present in `en.json`
- **THEN** each non-`en` catalog SHALL provide that key's value as a translation in its own language (not a verbatim English copy), with the parity test still green

### Requirement: Launcher overlay localization within the byte budget

The `Alt+L` launcher overlay SHALL display its user-facing strings localized within the overlay byte budget (`overlay.budget.test.ts`, <15KB), by one of two mechanisms: (Plan A) the
overlay imports the relevant `m.*` messages, permitted only if
`src/launcher/overlay.budget.test.ts` stays within its <15KB ceiling with those messages
tree-shaken in; or (Plan B) the service worker sends pre-localized label strings to the
overlay over the typed message bus, keeping the overlay catalog-free. The design SHALL
record the measured choice; whichever is chosen, the overlay budget guard SHALL stay
green and the overlay SHALL remain SW-safe (no `window`/`document` access for locale).

#### Scenario: The overlay budget guard holds after localization

- **WHEN** `overlay.budget.test.ts` runs after the overlay is localized
- **THEN** the overlay bundle SHALL remain under its 15KB ceiling

### Requirement: The pipeline fails on a new un-localized user-facing string

`pnpm verify` SHALL fail when a new user-facing string is introduced in a migrated
surface (`sidebar/`, `launcher/`, `options/`) as a literal instead of a message. The
mechanism SHALL be a Vitest guard `apps/extension/src/i18n-no-literal.test.ts` that parses
each surface `.svelte` template (via the Svelte compiler AST) and fails on a user-visible
literal text node, EXCEPT where allow-listed: whitespace/punctuation-only nodes, the brand
string "Lunma", text inside `code`/`pre`/`<style>`, legitimately-fixed attributes (class
names, `data-testid`, non-visible `aria` identifiers), and an explicit inline
`i18n-exempt` escape hatch (`<!-- i18n-exempt: reason -->` / `// i18n-exempt`). The guard
SHALL report the offending `file:line` and text. The guard SHALL cover template text
nodes AND user-facing attribute / component-prop literals (`placeholder`, `title`,
`aria-label`/`ariaLabel`, `alt`, `heading`, `label`, `description`, `subtitle`); literal
strings in `<script>` logic (e.g. option-list labels, toast text) are outside its scope
and remain a review concern.

A `m.*` reference with no catalog key SHALL fail `verify` — enforced by the TypeScript
compiler, not a separate lint: Paraglide generates a typed `m` namespace, so a missing key
is a compile error (`tsc`, already in `verify`), which is stronger than a runtime lint.

#### Scenario: A hardcoded surface literal fails verify

- **GIVEN** a developer adds a visible literal string to a migrated `.svelte` surface without a message key or an `i18n-exempt` marker
- **WHEN** `pnpm verify` runs
- **THEN** `i18n-no-literal.test.ts` SHALL fail, reporting the file, line, and the offending text

#### Scenario: An explicitly exempt literal passes

- **GIVEN** an intentional literal carries an inline `i18n-exempt` marker (or is the brand string "Lunma")
- **WHEN** the guard runs
- **THEN** it SHALL NOT fail on that literal

#### Scenario: A message reference with no catalog key fails verify

- **WHEN** a surface calls `m.someKey()` that is absent from `messages/en.json`
- **THEN** `tsc` (in `verify`) SHALL fail — the generated `m` namespace is typed, so the missing property is a compile error
